# spring-watch 采集层架构图

> 适用版本:基于 HertzBeat 调研结论改造后的 spring-watch
> 改造时间:2026-06-13
> 核心变化:由"全局 `@Scheduled` 拉所有应用" → **"每应用一个 `ScheduledFuture`,支持 interval 与 cron 双模式,热更新"**

---

## 一、问题与目标

### 改造前

```
@Scheduled(fixedDelay = 30s) {
    for (MonitorApp app : findAll()) {
        pull(app);     // ← 全局一把梭,所有 app 在同一时间窗
    }
}
```

问题:
- `scrapeInterval` 字段存了但**没生效**,所有应用共用一个固定周期
- 不支持 cron 表达式
- 改完周期需重启或被动等下个 tick,无热更新通道
- 无错峰,大量 app 同时拉 → 雪崩
- 限流粒度粗(没有)
- 单 app 拉取慢会阻塞后续 app

### 改造后

- 每应用一个调度任务,周期/cron 由 `monitor_app` 行决定
- `MonitorAppService` 的 `register/update/delete` 即刻同步到调度器,**热更新**
- 虚拟线程池 + per-host 限流 + jitter 错峰
- INTERVAL/CRON 双模式,坏 cron 自动回退

## 三、调度子系统内部组件关系

```mermaid
classDiagram
    class CollectScheduleRegistry {
        - ConcurrentHashMap~Long, ScheduledFuture~ jobs
        - ConcurrentHashMap~Long, AppSchedule~ activeSchedules
        - ScheduledExecutorService scheduler
        + run(ApplicationArguments) ApplicationRunner
        + upsert(MonitorApp) synchronized
        + cancel(Long appid)
        + activeJobCount() int
    }
    class AppSchedule {
        + Long appid
        + AppScheduleType type
        + long intervalMs
        + CronExpression cron
        + int jitterPercent
        + from(MonitorApp, int)$ AppSchedule
        + firstDelayMs(Instant) long
        + nextIntervalMs(Instant) long
    }
    class AppScheduleType {
        <<enum>>
        INTERVAL
        CRON
    }
    class AppScheduleProperties {
        + int poolSize = 32
        + int perHostConcurrent = 4
        + int jitterPercent = 10
    }
    class HostThrottler {
        - ConcurrentHashMap~String, Semaphore~ hostSemaphores

        + release(host)
    }
    class AppPullTask {
        + run(Long appid) @Transactional

        - extractHost(MonitorApp)
        - markInactive(MonitorApp, reason)
    }
    class MonitorApp {
        + Long appid
        + String appName
        + String endpoint
        + Integer metricsPort
        + Integer scrapeInterval
        + String scheduleType
        + String cronExpression
        + String status
    }

    CollectScheduleRegistry --> AppSchedule : 构造
    CollectScheduleRegistry --> AppScheduleProperties : 读配置
    CollectScheduleRegistry --> HostThrottler : 间接(经由 AppPullTask)
    CollectScheduleRegistry --> AppPullTask : 触发
    CollectScheduleRegistry --> MonitorApp : upsert 接收
    AppSchedule --> AppScheduleType : 持有
    AppSchedule --> MonitorApp : from() 解析
    AppPullTask --> HostThrottler : 限流
    AppPullTask --> MonitorApp : 拉取前查询
```

## 四、调度时序 - 单次采集周期(INTERVAL 模式)

```mermaid
sequenceDiagram
    autonumber
    participant VT as 虚拟线程池
    participant REG as CollectScheduleRegistry
    participant TASK as AppPullTask
    participant THR as HostThrottler
    participant MC as AgentMetricsCollector
    participant LC as AgentLogCollector
    participant KPB as KafkaProducerBridge
    participant DB as PostgreSQL
    participant AG as Agent (:8081)

    VT->>REG: 调度器触发<br/>safeRun(appid)
    REG->>TASK: run(appid)
    TASK->>DB: findByAppid(appid)
    DB-->>TASK: MonitorApp
    alt status != active
        TASK-->>REG: return(跳过)
    end
    TASK->>THR: tryAcquire(host, 5000ms)
    alt 获取失败
        THR-->>TASK: false
        TASK-->>REG: return(限流跳过,记 warn)
    else 获取成功
        TASK->>AG: HEAD/GET http://host:9464/metrics<br/>(isReachable)
        alt 不可达
            TASK->>DB: update status=inactive
            TASK-->>THR: release
            TASK-->>REG: return
        end
        TASK->>MC: collect(target)
        MC->>AG: GET /metrics
        AG-->>MC: Prometheus 文本
        MC->>KPB: sendMetric × N
        TASK->>KPB: sendHeartbeat
        TASK->>LC: collect(since=lastLogPullTime)
        LC->>AG: GET /api/agent/logs?since=
        AG-->>LC: JSON logs
        LC->>KPB: sendLog × N
        opt latest > since
            TASK->>DB: update lastLogPullTime
        end
        TASK->>THR: release(host)
    end
    REG->>VT: scheduleWithFixedDelay 重新入队
```

## 五、调度时序 - 热更新(用户修改 scrapeInterval / cron)

```mermaid
sequenceDiagram
    autonumber
    participant U as 用户/UI
    participant CTRL as MonitorAppController
    participant SVC as MonitorAppService
    participant DB as PostgreSQL
    participant REG as CollectScheduleRegistry
    participant VT as 虚拟线程池
    participant TASK as AppPullTask

    U->>CTRL: PUT /api/apps/{id}<br/>{scrapeInterval: 60, scheduleType: "CRON",<br/>cronExpression: "0 */1 * * * *"}
    CTRL->>SVC: update(id, request)
    SVC->>DB: findById(id)
    DB-->>SVC: 旧 MonitorApp
    SVC->>SVC: setScrapeInterval / setScheduleType / setCronExpression
    SVC->>DB: save(新 MonitorApp)
    DB-->>SVC: saved
    SVC->>REG: upsert(saved)
    REG->>VT: 找到旧 ScheduledFuture
    REG->>VT: cancel(false) 旧 future
    REG->>REG: cancelInternal(appid)
    REG->>REG: 重新计算 AppSchedule.from(app)
    alt type == CRON
        REG->>VT: schedule(cronLoop, firstDelayMs)
    else type == INTERVAL
        REG->>VT: scheduleWithFixedDelay(safeRun, jitter/2, interval)
    end
    REG->>SVC: return
    SVC-->>CTRL: saved
    CTRL-->>U: 200 OK

    Note over VT,TASK: 下一次采集即按新配置执行<br/>无需重启
```

## 六、调度时序 - CRON 模式循环

```mermaid
sequenceDiagram
    autonumber
    participant VT as 虚拟线程池
    participant REG as CollectScheduleRegistry
    participant CRON as Spring CronExpression
    participant TASK as AppPullTask

    REG->>VT: schedule(() -> cronLoop(appid), firstDelayMs)
    VT-->>REG: 触发(到点)
    REG->>TASK: safeRun(appid)
    TASK-->>REG: 采集完成
    REG->>CRON: cron.next(now)
    CRON-->>REG: 下个触发时刻
    REG->>VT: schedule(() -> cronLoop(appid), nextDelayMs)
    VT-->>REG: 触发(到点)
    REG->>TASK: safeRun(appid)
    Note over REG: 若中途 upsert/cancel,<br/>activeSchedules / jobs 变更,<br/>下一次进入 cronLoop 时发现<br/>jobs.get(appid) 已变,自动退出
```

## 七、关键设计点对照 HertzBeat 经验

| 设计点 | HertzBeat 经验 | spring-watch 落地 |
|---|---|---|
| 调度模型 | 每 Monitor 一个时间轮槽 | 每 app 一个 `ScheduledFuture` |
| 时间结构 | Dubbo `HashedWheelTimer` | `ScheduledExecutorService` + 虚拟线程 |
| 执行线程 | Java 21 虚拟线程 | Java 25 虚拟线程(`Thread.ofVirtual().factory()`) |
| 周期策略 | INTERVAL 或 CRON,二选一 | 同上,坏 CRON 自动回退 INTERVAL |
| 错峰 | HertzBeat 用 LCM 差分(无 jitter) | **`±jitterPercent` 随机抖动**,启动时防雪崩 |
| 限流 | 全局 `Semaphore(512)` | **per-host `Semaphore(perHostConcurrent)`** |
| 热更新 | delete + add 重建 | delete + add 重建,加 `activeSchedules` 检测防止僵尸 future |
| 启停 | `SchedulerInit.run()` + `reBalance` | `ApplicationRunner.run()`(启动加载) + `@PreDestroy`(优雅关闭) |
| 配置 | `@Value` 硬编码 | `@ConfigurationProperties` + yml,`pool-size/per-host/jitter` 可调 |
