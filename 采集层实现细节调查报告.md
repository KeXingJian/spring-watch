# HertzBeat 采集层实现细节调查报告

> 项目根:`D:\codespace\ideaProject\hertzbeat`
> 调研时间:2026-06-13
> 调研目的:分析 HertzBeat 采集层实现细节,回答"目前是定时采集所有应用还是可配置单应用"的问题,并评估其对大量应用采集的优化情况。

---

## 一、总体结论

| 问题 | 结论 |
|---|---|
| 目前是"定时采集所有应用"还是"每应用可配置"? | **每 Monitor 一个调度任务、周期可配置**(在数据模型和调度层都已具备) |
| 调度实现? | **不是** `@Scheduled` / Quartz,而是 **Dubbo `HashedWheelTimer` 时间轮** + **Java 21 虚拟线程池** |
| 配置变更是否动态生效? | **是**,但实现是"删旧 Job + 建新 Job",会有 1~2 个周期空缺 |
| 是否有大量应用采集优化? | **部分有**(连接池、LCM 错峰、优先级队列、连接 LRU、Micrometer 指标、Apache Arrow 序列化),**但缺关键优化**(无 jitter 抖动、Manager 单点、限流粒度粗、不合并同目标) |

  ```java
  public Long getNextExecutionInterval(Job job) {
      if (ScheduleTypeEnum.CRON.getType().equals(job.getScheduleType()) && job.getCronExpression() != null && ...) {
          CronExpression cronExpression = CronExpression.parse(job.getCronExpression());
          ZonedDateTime nextExecutionTime = cronExpression.next(ZonedDateTime.now());
          long delay = Duration.between(ZonedDateTime.now(), nextExecutionTime).toMillis();
          return Math.max(0, delay / 1000);
      } else {
          if (job.getDispatchTime() > 0) {
              long spendTime = System.currentTimeMillis() - job.getDispatchTime();
              long intervalMs = job.getInterval() * 1000 - spendTime;
              return Math.max(0, intervalMs / 1000);
          }
          return job.getInterval();
      }
  }
  ```
- **特殊点**:不是简单"interval 秒",而是先 `initIntervals()` 计算所有 metrics 间隔的 LCM 列表(轮转式)—— `Job.java:298-363`
  ```java
  public synchronized void generateMetricsIntervals(List<Long> metricsIntervals) {
      long lcm = lcm(metricsIntervals);
      List<Long> refreshTimes = new LinkedList<>();
      for (long interval : metricsIntervals) {
          for (long t = interval; t <= lcm; t += interval) {
              if (!refreshTimes.contains(t)) refreshTimes.add(t);
          }
      }
      Collections.sort(refreshTimes);
      LinkedList<Long> intervals = new LinkedList<>();
      intervals.add(refreshTimes.get(0));
      for (int i = 1; i < refreshTimes.size(); i++) {
          intervals.add(refreshTimes.get(i) - refreshTimes.get(i - 1));   // 差分!
      }
      setIntervals(new ConcurrentLinkedDeque<>(intervals));
  }
  ```
  - `getInterval()` 用 round-robin 取差分队列 —— `Job.java:365-374`

### 2.3 调度时如何加载所有需采集的 Monitor

**启动时一次性全量加载,缓存到 Manager 端 `jobContentCache`(`ConcurrentHashMap<Long, Job>`)+ Collector 端 `currentCyclicTaskMap`,不再轮询 DB。**

- 启动全量:`SchedulerInit.java:97-99` SQL 一次 `findMonitorsByStatusNotInAndJobIdNotNull`
- 运行时变更:通过 `MonitorService.addMonitor / modifyMonitor / deleteMonitor` 主动调用 `addAsyncCollectJob / updateAsyncCollectJob / cancelAsyncCollectJob` 增删任务,**不经轮询**
- 缓存:`CollectorJobScheduler.java:70`
  ```java

  ```
- 监听器缓存:`CollectorJobScheduler.java:72` —— `eventListeners`(同步任务等待响应用)

### 2.4 是否使用分布式调度

**是,基于"Manager + 一致性 Hash Collector 集群",Manager 本身不分布式。**

- 一致性 Hash 路由:`ConsistentHash.java:40-66`
  - FNV1_32_HASH 算法,默认每个 Collector 10 个虚拟节点(`VIRTUAL_NODE_DEFAULT_SIZE = 10`)
  - 调度 key = `String.valueOf(job.getMonitorId())`(即 `monitorId`)
- 任务分配数据结构:`AssignJobs.java:30-58` —— 每个 collector 节点维护 4 个 Set:
  - `jobs` —— 当前已分配
  - `addingJobs` —— 待下发
  - `removingJobs` —— 待取消
  - `pinnedJobs` —— 钉死(指定 collector)
- Collector 上下线/负载变化触发 **reBalance**:`CollectorJobScheduler.java:201-250` `reBalanceCollectorAssignJobs()`,使用 `parallelStream()` 并行处理
- 集群通信:Netty RPC
  - **Manager 端**:`ManageServer.java`(`hertzbeat-manager/.../scheduler/netty/ManageServer.java:60`)
  - **Collector 端**:`CollectServer.java`(`hertzbeat-collector/.../dispatch/entrance/CollectServer.java:66`),端口默认 1158
  - 序列化:Protobuf(`ClusterMsg.Message`)
  - 消息类型:`ISSUE_CYCLIC_TASK` / `DELETE_CYCLIC_TASK` / `ISSUE_ONE_TIME_TASK` / `RESPONSE_CYCLIC_TASK_DATA` / `GO_ONLINE` / `GO_OFFLINE` / `HEARTBEAT` 等
- 心跳/健康检查:`ManageServer.java:134, 206-319` —— 每 3 秒检查一次 channel 是否 active,断线触发 `collectorGoOffline()`

## 三、采集执行器分析(Collector 端)

### 3.1 调度线程池大小

**使用 Java 21 虚拟线程(`VirtualThread`)池 + 限流信号量,默认 `LIMIT_AND_REJECT` 模式上限 512。**

- **`WorkerPool`** —— `hertzbeat-collector/.../dispatch/WorkerPool.java:38-125`
- 默认配置(`VirtualThreadProperties.PoolProperties.collectorDefaults()`)—— `VirtualThreadProperties.java:34, 79-81`
  ```java
  private static final int DEFAULT_COLLECTOR_MAX_CONCURRENT_JOBS = 512;
  public static PoolProperties collectorDefaults() {
      return new PoolProperties(AdmissionMode.LIMIT_AND_REJECT, defaultCollectorConcurrency());
  }
  ```
- 支持 3 种 `AdmissionMode`(`AdmissionMode.java:23-39`):
  - `UNBOUNDED_VT` —— 无限虚拟线程
  - `LIMIT_AND_REJECT` —— 限流,超限直接 `RejectedExecutionException`
  - `LIMIT_AND_BLOCK` —— 限流,提交方阻塞
- 默认非虚拟线程 fallback(`WorkerPool.java:82-97`):
  ```java
  int coreSize = Math.max(2, Runtime.getRuntime().availableProcessors());
  int maxSize = Runtime.getRuntime().availableProcessors() * 16;
  return new ThreadPoolExecutor(coreSize, maxSize, 10, TimeUnit.SECONDS,
          new SynchronousQueue<>(), threadFactory, new ThreadPoolExecutor.AbortPolicy());
  ```
- 配置文件 `application.yml:351-379`:
  ```yaml
  hertzbeat:
    vthreads:
      enabled: true
      common:
        mode: UNBOUNDED_VT
      collector:
        mode: LIMIT_AND_REJECT
      manager:
        mode: LIMIT_AND_REJECT
        max-concurrent-jobs: 10
  ```
  - 注意:`collector` 没有显式 `max-concurrent-jobs`,沿用默认 512

### 3.2 采集任务如何提交(串行/并行)

**单 dispatcher 线程 + 优先级队列 + 虚拟线程池并行执行,metrics 级并行,metric 内串行。**

3. **单 dispatcher 线程消费**:`CommonDispatcher.java:114-140`

### 3.3 是否做了并发限流 / 优先级 / 分片

#### 限流
- 通过 `AdmissionMode.LIMIT_AND_REJECT` 在虚拟线程池入口用 Semaphore 限流(`ManagedExecutors.java:60-68, 192-214`)
- **限流后行为**:`CommonDispatcher.java:126-131` —— 拒绝后将该任务的 `runPriority++`(降级)并重投入队
- **⚠️ 不足**:O(1) 简单方案,缺乏"按 collector 节点限流"、"按目标主机限流"、"按 app 类型限流"等细粒度策略

#### 优先级
- 三档:-1(周期)、1(一次性),降级时 +1
- 也支持 metric 级别优先级(0-127):`Job.java:165-175`
  ```java
  /**
   * 0 - availability
   * 1 - cpu | memory
   * 2 - health
   * 3 - otherMetrics
   * 127 - lastPriorMetrics
   */
  private transient LinkedList<Set<Metrics>> priorMetrics;
  ```
- 任务按 `priority` 分组,串行执行 —— `Job.constructPriorMetrics()` —— `Job.java:186-228`

#### 分片
- **集群级分片**:一致性 Hash(`ConsistentHash`)
- **Job 内分片(子任务)**:用于 `cryPlaceholder` 占位符展开后的并行采 —— `CommonDispatcher.java:269-300`
  ```java
  int subTaskNum = isSubTask ? Math.min(configmapList.size(), MAX_SUB_TASK_NUM) : 1;  // MAX=50
  ```
- **⚠️ 不足**:`MAX_SUB_TASK_NUM = 50`(硬编码)(`CommonDispatcher.java:66`),不能配置;且只在 cry 占位符场景下生效,普通监控不会自动分片

#### 时间轮
- 使用 **Dubbo `HashedWheelTimer`**(`hertzbeat-common-core/.../timer/HashedWheelTimer.java:83`)
  - `TimerDispatcher.java:89-93` 构造:
    ```java
    new HashedWheelTimer(r -> new Thread(r, "wheelTimer"), 1, TimeUnit.SECONDS, 512);
    ```
  - tick = 1 秒,512 槽
  - 派生自 Dubbo 3.2 源码(`HashedWheelTimer.java:40-41` 注释)

### 3.4 实际采集执行

- **`MetricsCollect.run()`** —— `MetricsCollect.java:154-226`
  - 选择协议实现:`CollectStrategyFactory.invoke(metrics.getProtocol())`(SPI 加载,见 §六)
  - 协议相关实现位于 `hertzbeat-collector-basic/.../collect/{http,snmp,ssh,jmx,redis,...}/`

### 4.2 配置变更后是否动态生效(热更新)

**部分支持,但实现是"删除+重建",不是"原地修改"——会丢失当前正在执行的采集。**

- 修改流程:`MonitorServiceImpl.modifyMonitor()`(`MonitorServiceImpl.java:360-440`)
  1. 校验
  2. 构造新的 `Job`(含新 `defaultInterval`)
  3. 调用 `collectJobScheduling.updateAsyncCollectJob(appDefine)` → 实际是 `addAsyncCollectJob` + `cancelAsyncCollectJob(preJobId)`(`CollectorJobScheduler.java:425-432`)
  4. 远端 collector 走 Netty 通知:`CollectorJobScheduler.java:413-419` 发 `ISSUE_CYCLIC_TASK`,`CollectorJobScheduler.java:461-466` 发 `DELETE_CYCLIC_TASK`
- **实际效果**:
  - 旧 task 在时间轮中被 cancel(`TimerDispatcher.deleteJob` —— `TimerDispatcher.java:155-167`)
  - 新 task 重新入时间轮
  - 正在执行中的旧 `MetricsCollect` 不受影响(Java 线程无法打断),但下一周期会用新配置
- 删除/暂停/取消:
  - `cancelAsyncCollectJob` 移除本地缓存 + 通知远端 —— `CollectorJobScheduler.java:444-474`
  - 暂停状态(`status = 0`)在启动时就被 `findMonitorsByStatusNotInAndJobIdNotNull` 过滤掉

### 4.3 调度器如何感知周期变化

**无轮询、无事件订阅,完全靠应用层主动 push。** 修改/新增/删除 monitor 的 Controller → Service 层同步调用 `CollectJobScheduling` 的 `add/update/cancelAsyncCollectJob` 方法,经 Netty 下发到 collector;collector 端 `CollectCyclicDataProcessor` 接收(`hertzbeat-collector/.../entrance/processor/CollectCyclicDataProcessor.java:40-48`):

```java
public ClusterMsg.Message handle(ChannelHandlerContext ctx, ClusterMsg.Message message) {
    Job job = JsonUtil.fromJson(message.getMsg().toStringUtf8(), Job.class);
    if (job == null) {
        log.error("collector receive cyclic task job is null");
        return null;
    }
    collectServer.getCollectJobService().addAsyncCollectJob(job);
    return null;
}
```

- **⚠️ 不足**:无 mq / 无 etcd watch,Manager 单点是瓶颈;若 manager 宕机期间有人改库,重启后才会被 `SchedulerInit.run()` 重新加载;运行中配置变更靠 API 同步,没有"重载全部"机制

## 六、大量应用采集优化盘点

### 6.1 已具备的优化

| 优化手段 | 状态 | 位置 |
|---|---|---|
| **错峰 / 错开执行** | ✅ 部分 | `Job.generateMetricsIntervals()` 计算 LCM 差分,保证不同 monitor 的采集时刻不同 |
| **批量加载 app 定义** | ✅ | `AppServiceImpl.appDefines`(`AppServiceImpl.java:100`)—— `ConcurrentHashMap<String, Job>` 缓存 yml 解析结果 |
| **采集结果队列缓冲** | ✅ | `CommonDataQueue`(`InMemoryCommonDataQueue.java:48-172`)—— 5 个 `LinkedBlockingQueue`(metricsToAlert / metricsToStorage / sd / log) |
| **日志批量入库** | ✅ | `DataStorageDispatch.java:120, 145` —— `pollLogEntryToStorageBatch(maxBatchSize)` 批量 poll |
| **Warehouse 单消费者线程** | ✅ | `DataStorageDispatch.java:74-111` —— 单一长跑线程轮询持久化队列,背压指数退避(`ExponentialBackoff`) |
| **HTTP 连接池** | ✅ | `CommonHttpClient.java:67-72` —— 50,000 总连接,80/路由,40 秒清理 |
| **全局连接 LRU 缓存** | ✅ | `GlobalConnectionCache.java:37-303` —— `ConcurrentLinkedHashMap`,10 分钟过期,100 秒扫描清理,虚拟线程清理 |
| **SubTask 拆并行** | ⚠️ 仅 cryPlaceholder | `CommonDispatcher.MAX_SUB_TASK_NUM = 50` |
| **同 monitor 的多 metric 并行** | ❌ 串行 | `Job.constructPriorMetrics` 按 priority 顺序 poll |
| **错峰(jitter)** | ❌ 无 | 入时间轮时直接 `interval` 秒,无随机抖动,易出现"雪崩" |
| **合并多个 monitor 同一目标** | ❌ 无 | 每个 monitor 独立 job,即使 IP+协议相同也不会合并 |
| **Caffeine** | ⚠️ 仅在 AccountService/DataSource | 采集主链路未使用 |

## 七、关键问题与不足总结

| 方面 | 当前实现 | 关键不足 |
|---|---|---|
| **调度模型** | 每 Monitor 一个时间轮任务 + LCM 差分 | 无错峰抖动,易雪崩;重启后批量加载,运行中变更需手动 push |
| **优先级队列** | `PriorityBlockingQueue<MetricsCollect>`,周期 -1/一次 +1 | 周期任务 `runPriority=-1` 反而**被排在队尾**(因取反);降级时 +1 不一定能跑到 0/1 之前 |
| **线程池** | Java 21 虚拟线程 + LIMIT_AND_REJECT(默认 512) | 限流粒度粗(全局),不能按 app/host/collector 限流;Reentrant 失败后仅简单重投,无 dead letter |
| **配置热更新** | delete + add 重建 | 正在执行的 metrics 不能被中断,可能有 1-2 周期空缺;Manager 不支持多实例 |
| **分布式调度** | Manager 单点 + 多 Collector(一致性 Hash) | Manager 单点瓶颈;无 leader election;无任务抢占/转移 |
| **超时监控** | 240 秒硬编码 | `DURATION_TIME` 不可配置;扫描间隔 20 秒,长尾任务清理慢 |
| **数据缓冲** | LinkedBlockingQueue(内存) | 无背压丢弃策略;无磁盘溢写;Kafka 模式需要外置 |
| **连接池** | HTTP 50000 总,80/route(硬编码) | 不能按 monitor 限速;SSH/JDBC/Redis 走 GlobalConnectionCache 但 `maximumWeightedCapacity=Integer.MAX_VALUE` 易 OOM |
| **Caffeine** | 仅辅助服务用,采集主链路无 | 解析的 YML/AppDefine 无 Caffeine,而是 ConcurrentHashMap,虽然够用但无 LRU/TTL |
| **指标统计** | Micrometer Counter + Timer | 无 P99/HdrHistogram;无每秒采集次数(QPS)指标;无丢弃率/超时率维度 |
| **插件** | Java SPI + ServiceLoader(协议级)+ URLClassLoader jar(钩子级) | 运行时 jar 插件只能挂 post 钩子,**不能新增协议采集器** |
| **错峰** | 仅 LCM 差分序列 | 无随机 jitter;无 off-peak 调度;无 cron `H` 字段式分散 |
| **合并采集** | 无 | 同 IP+协议的不同 monitor 各自独立采集 |
| **失败处理** | metric 0 失败 → 短路,直接下一轮 | 没有"重试 N 次"机制;没有 backoff;部分成功/失败无统计 |
| **数据回写** | 单线程消费队列 + 指数退避 | 单点吞吐,history tsdb 写慢时会阻塞 alert |
| **租户隔离** | `tenantId` 字段已存在 | 调度层面不区分租户,无独立队列/线程隔离 |

## 八、改造方向建议

如果你的目标是"**每个应用可配周期 + 热更新**":

- **HertzBeat 本身已支持**,API 层调用 `MonitorController.modifyMonitor` 即可,核心改动在 `MonitorServiceImpl` 做一层"按 host / 按 tenant 分桶调度"

如果规模到 **10w+ monitor**,按优先级:

1. **加随机 jitter**(`Job` 构造时 `defaultInterval + ThreadLocalRandom.nextInt(0, interval/10)`)
2. **按目标主机分桶限流**(`Semaphore` 按 `host:port`)
3. **同主机同协议采集合并**(在 `Job` 维度做 dedup,共用一次 HTTP 请求多 metric)
4. **Caffeine 缓存 AppDefine + 指标历史**(替代 `ConcurrentHashMap`,加 TTL/最大容量)
5. **将 Manager 调度层独立出来做集群**(`SchedulerInit` + `CollectJobScheduling` 拆出,接 etcd/Nacos 选主)
6. **Micrometer 加 DistributionSummary**,补 P50/P99
