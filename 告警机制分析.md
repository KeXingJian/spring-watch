# HertzBeat 告警机制分析

> 目标项目: `D:\codespace\ideaProject\hertzbeat`
> 分析范围: 告警规则配置、指标/日志数据匹配告警、告警行为分发
> 涉及模块: `hertzbeat-alerter`(核心), `hertzbeat-common-spring`(实体/常量), `hertzbeat-warehouse`(数据源)

---

## 0. 总体架构

```
+------------------------------------------------------------------+
|                       数据采集层 (Collector)                     |
|        抓取指标 / 接收日志 → 入库 + 入队 CommonDataQueue         |
+------------------------------------------------------------------+
                              │
                              ▼ (Redis/Kafka/InMemory 队列)
+------------------------------------------------------------------+
|                 告警计算层 (hertzbeat-alerter.calculate)         |
|                                                                  |
|  实时指标: MetricsRealTimeAlertCalculator   ←─── 3 消费线程      |
|  实时日志: WindowedLogRealTimeAlertCalculator + LogWorker         |
|             + WindowAggregator + AlarmEvaluator                  |
|  周期指标: PeriodicAlertRuleScheduler → MetricsPeriodicAlertCalc  |
|  周期日志: PeriodicAlertRuleScheduler → LogPeriodicAlertCalc     |
|  采集器:   CollectorAlertHandler (上下线)                        |
+------------------------------------------------------------------+
                              │ 命中阈值
                              ▼
              AlarmCommonReduce (异步 Reduce 线程池)
                              │
                              ▼
              AlarmGroupReduce (按规则分组 + 抑制 + 等待/发送策略)
                              │
                              ▼
              AlarmInhibitReduce (抑制规则过滤)
                              │
                              ▼
              AlarmSilenceReduce (静默规则过滤)
                              │
                              ▼
              AlertNoticeDispatch (存储 + 通知分发)
                 │           │           │
                 ▼           ▼           ▼
           DbAlertStore    通知发送器    SSE 广播
           (落库)        (Email/IM…)   (web 推送)
```

告警行为遵循"采集→计算→Reduce→分发"四阶段管道,每阶段都有独立的可配置策略。

## 1. 告警是怎么做的(规则模型与数据流)

### 1.1 核心实体

实体定义位于 `hertzbeat-common-spring/src/main/java/org/apache/hertzbeat/common/entity/alerter/`。

| 实体 | 表 | 关键字段 | 作用 |
|------|----|----------|------|
| `AlertDefine` | `hzb_alert_define` | `name`、`type`、`expr`、`period`、`times`、`labels`、`annotations`、`template`、`datasource`、`enable` | **告警阈值规则**(核心) |
| `AlertSilence` | `hzb_alert_silence` | `name`、`enable`、`matchAll`、`type`(0单次/1周期)、`times`、`labels`、`days`、`periodStart`、`periodEnd` | 静默策略 |
| `AlertInhibit` | `hzb_alert_inhibit` | `name`、`sourceLabels`、`targetLabels`、`equalLabels`、`enable` | 抑制策略(高级别告警屏蔽低级别) |
| `AlertGroupConverge` | `hzb_alert_group_converge` | `name`、`groupLabels`、`groupWait`、`groupInterval`、`repeatInterval`、`enable` | 分组收敛策略 |
| `NoticeRule` | `hzb_notice_rule` | `name`、`receiverId[]`、`templateId`、`enable`、`filterAll`、`labels`、`days`、`periodStart`、`periodEnd` | 通知策略 |
| `NoticeReceiver` | `hzb_notice_receiver` | `name`、`type`(0~14 通道)、各通道字段(phone/email/hookUrl/wechatId/tgBotToken 等) | 通知接收人 |
| `NoticeTemplate` | `hzb_notice_template` | `name`、`type`、`preset`、`content`(Freemarker 模板) | 通知模板(自定义) |
| `SingleAlert` | `hzb_alert_single` | `fingerprint`、`labels`、`annotations`、`content`、`status`、`triggerTimes`、`startAt/activeAt/endAt` | 单条告警 |
| `GroupAlert` | `hzb_alert_group` | `groupKey`、`groupLabels`、`commonLabels`、`commonAnnotations`、`alertFingerprints[]`、`alerts[]` | 分组告警 |

> `SingleAlert` 和 `GroupAlert` 是告警计算产出的运行时对象,前者落库(`hzb_alert_single`,唯一索引 `fingerprint`),后者以 `groupKey` 为主键落 `hzb_alert_group`。

### 1.2 规则类型

`AlertDefine.type` 字段决定计算入口(`AlertDefine.java:78`),取自 `CommonConstants`(`hertzbeat-common-core/.../constants/CommonConstants.java:153-168`):

| type 常量 | 值 | 含义 |
|----------|----|------|
| `METRIC_ALERT_THRESHOLD_TYPE_REALTIME` | `realtime_metric` | 实时指标规则:消费队列里的指标,逐条 JEXL 评估 |
| `METRIC_ALERT_THRESHOLD_TYPE_PERIODIC` | `periodic_metric` | 周期指标规则:定时拉数据源(支持 PromQL/SQL),ANTLR 解析后下发 |
| `LOG_ALERT_THRESHOLD_TYPE_REALTIME` | `realtime_log` | 实时日志规则:消费日志流,JEXL 评估,按时间窗聚合 |
| `LOG_ALERT_THRESHOLD_TYPE_PERIODIC` | `periodic_log` | 周期日志规则:定时跑 SQL,直接产出 |

### 1.3 表达式与模板

- **指标/日志实时表达式**:JEXL(`org.apache.commons.jexl3`)
  - 入口: `JexlExpressionRunner.compile/evaluate`
  - 计算器: `alert/calculate/JexlExprCalculator.java:43`
  - 特性: 关闭了 loops/lambda/newInstance/methodCall 等能力,只允许 JEXL 内置函数 + 自定义 `json:apply(x)` 函数(`JexlExpressionRunner.java:55-60`)
- **周期指标表达式**:ANTLR4 语法
  - 语法文件: `hertzbeat-alerter/src/main/resources/expr/AlertExpression.g4`
  - 解析器: `DataSourceServiceImpl.calculate()` 缓存 ParseTree(`DataSourceServiceImpl.java:69-73`)
  - 支持 PromQL 子集(`metricSelector`、聚合、`rate`/`increase`/`histogram_quantile` 等)与 SQL 子集(`SELECT ... FROM ... WHERE ... GROUP BY ... HAVING ... ORDER BY ... LIMIT`)
  - SQL 数据源额外走 `SqlSecurityValidator`(`DataSourceServiceImpl.java:145-152`)
- **告警内容模板**: `AlertTemplateUtil.render()`
  - 语法: `${key}` 或 `${key.subKey.field}`(支持嵌套 Map/POJO 反射)
  - 用法示例(在 `AlertDefine.template`):`Instance ${instance} CPU usage is ${usage}%`
  - 通知模板(`NoticeTemplate.content`)则使用 **Freemarker**(`AbstractAlertNotifyHandlerImpl.renderContent()` 注入 `title/status/groupLabels/commonLabels/commonAnnotations/alerts/consoleUrl`)

### 1.4 指纹(Fingerprint)机制

`AlertUtil.calculateFingerprint()` (`alert/util/AlertUtil.java:34-39`): 把 `labels` 排序后拼接为 `hash(keys)-hash(values)`。用于:
- 实时计算中**消重**(`pendingAlertMap` / `firingAlertMap`,主键为 `defineId + fingerprint`)
- 持久化时**唯一索引**(`hzb_alert_single.fingerprint` 唯一)
- 同一指标的多次上报在缓存层被合并,实现"按 N 次触发后才告警"

## 3. 指标数据怎么匹配告警

### 3.1 实时指标(`realtime_metric`)

入口: `alert/calculate/realtime/MetricsRealTimeAlertCalculator.java`

1. **取数据**:`dataQueue.pollMetricsDataToAlerter()`(从 `CommonDataQueue` 拉,失败走指数退避)
2. **筛选规则**:`filterThresholdsByAppAndMetrics(...)` 用正则解析 expr 中的 `equals(__app__, "...")`,再按 `__instance__`/`__labels__` 匹配当前指标行(`MetricsRealTimeAlertCalculator.java:301-357`)
3. **构造上下文**:`__instance__`、`__instancename__`、`__instancehost__`、`__app__`、`__metrics__`、`__priority__`、`__code__`、`__available__`、所有 labels
4. **JEXL 评估**:`jexlExprCalculator.execAlertExpression(fieldValueMap, expr, ...)`(先按"全局上下文"试一次,失败再按"行级"逐列试)
5. **匹配后处理**:`afterThresholdRuleMatch()`
   - 计算 `fingerprint` = `AlertUtil.calculateFingerprint(fingerPrints)`(`fingerPrints` 包含 `defineId` + `alertname` + 指标 labels)
   - 在 `pendingAlertMap` 中查找;无则**新建** SingleAlert(`status=PENDING`、`triggerTimes=1`)
   - 若 `define.times==1` → 直接置 `FIRING`,加入 `firingAlertMap`,**送入 `AlarmCommonReduce`**
   - 否则只在 `pendingAlertMap` 中累加 `triggerTimes`
   - 已存在 pending 的 → 累加;达到 `times` 才升级 `FIRING` 并下发
6. **未匹配**:`handleRecoveredAlert()` — 把 `firingAlertMap` 中的同 fingerprint 移除并以 `status=RESOLVED` 下发(`MetricsRealTimeAlertCalculator.java:362-373`)
7. **缓存容器**:`AlarmCacheManager`
   - `pendingAlertMap`: `Table<defineId, fingerprint, SingleAlert>`(等待达到 N 次)
   - `firingAlertMap`: `Table<defineId, fingerprint, SingleAlert>`(已 firing,等待恢复)
   - 启动时从 DB 加载 `ALERT_STATUS_FIRING` 状态的告警以恢复上下文(`AlarmCacheManager.java:54-67`)

### 3.2 周期指标(`periodic_metric`)

入口: `alert/calculate/periodic/PeriodicAlertRuleScheduler.java` + `MetricsPeriodicAlertCalculator.java`

1. **调度**:`PeriodicAlertRuleScheduler` 是 `CommandLineRunner`,启动时按 `define.period` 用 `scheduledExecutor.scheduleAtFixedRate` 给每条 `periodic_metric` / `periodic_log` 规则注册定时任务
2. **配置变更**:`AlertDefineServiceImpl` 增删改规则后调用 `updateSchedule(rule)` / `cancelSchedule(ruleId)` 重新调度
3. **执行**:`MetricsPeriodicAlertCalculator.calculate(rule)` 调用 `dataSourceService.calculate(datasource, expr)`(`DataSourceServiceImpl.calculate`)
   - ANTLR 解析 → 走对应 `QueryExecutor`(PromQL/SQL)
   - 返回 `List<Map<String, Object>>`,约定结果中 `__value__` = 命中值,`__timestamp__` = 时间,其他字段为标签
4. **遍历结果**:每行生成 `fingerPrints`(行字段作为指纹),`__value__==null` 视为"已恢复",调用 `handleRecoveredAlert`;否则 `afterThresholdRuleMatch` 走与实时一致的 pending/firing 流程(`MetricsPeriodicAlertCalculator.java:76-184`)

### 3.3 实时日志(`realtime_log`)

入口: `WindowedLogRealTimeAlertCalculator` → `LogWorker` → `WindowAggregator` → `AlarmEvaluator`

- **拉取日志**: `dataQueue.pollLogEntry()`,抽取 `timeUnixNano/observedTimeUnixNano` 作为事件时间,丢弃迟于 watermark 的数据,3 个 dispatcher 线程并发(`WindowedLogRealTimeAlertCalculator.java:64-107`)
- **表达式评估**: 上下文 `{"log": LogEntry}`, JEXL 评估所有 `realtime_log` 规则(`LogWorker.java:74-101`);命中后投递 `MatchingLogEvent` 到 `WindowAggregator`
- **窗口聚合**: `WindowAggregator` 按 `define.period` 秒划分时间窗,定时关闭并把窗口内匹配日志交给 `AlarmEvaluator`
- **生成告警**:
  - 累加 `triggerTimes`(`define.times` 默认 1)
  - 达到阈值后根据 `define.labels["alert_mode"]`(`group`/`individual`)生成 SingleAlert 列表(`AlarmEvaluator.java:99-137`)

### 3.4 周期日志(`periodic_log`)

入口: `LogPeriodicAlertCalculator.calculate(rule)`(同 `MetricsPeriodicAlertCalculator`,但走 `dataSourceService.query` 走纯 SQL)
- 结果逐行 → `generateIndividualAlert`(每条都发)或 `generateGroupAlert`(打成一个分组告警)(`LogPeriodicAlertCalculator.java:95-189`)
- 模式由 `define.labels[ALERT_MODE_LABEL]` 决定,默认 `group`

### 3.5 采集器可用性告警

`CollectorAlertHandler`: 不走规则,而是监听 collector 上/下线事件直接 `FIRING`/`RESOLVED`,内容取 `alerter_*.properties` 国际化文案(`CollectorAlertHandler.java:69-122`)。

## 4. 告警行为(通知)怎么配置与触发

### 4.1 三类"行为"策略

| 策略 | 实体 | 触发点 | 文件 |
|------|------|--------|------|
| **分组收敛** | `AlertGroupConverge` | 告警进入 `AlarmGroupReduce` | `alert/reduce/AlarmGroupReduce.java` |
| **抑制** | `AlertInhibit` | 告警通过分组后 | `alert/reduce/AlarmInhibitReduce.java` |
| **静默** | `AlertSilence` | 通过抑制后 | `alert/reduce/AlarmSilenceReduce.java` |

四阶段 Reduce 链:
```
AlarmCommonReduce.reduceAndSendAlarm(alert)        ← 入口(异步)
   └─> AlarmGroupReduce.processGroupAlert(...)      ← 按 group_labels 缓存/聚合/发送
        └─> AlarmInhibitReduce.inhibitAlarm(...)    ← sourceLabels 命中则屏蔽 targetLabels
             └─> AlarmSilenceReduce.silenceAlarm(...) ← 静默规则过滤
                  └─> AlertNoticeDispatch.dispatchAlarm(groupAlert) ← 真发
```

> 注: `AlarmInhibitReduce` / `AlarmSilenceReduce` 的设计参考 Prometheus Alertmanager。`AlarmGroupReduce` 与 Prometheus 的 `group_wait / group_interval / repeat_interval` 概念一致。

### 4.2 分组收敛 `AlertGroupConverge`

字段: `groupLabels[]`、`groupWait`(秒)、`groupInterval`(秒)、`repeatInterval`(秒)、`enable`(`AlarmGroupConverge.java`)

行为(`AlarmGroupReduce.java`):
- 用 `groupLabels` 抽取每条告警的"分组指纹",生成 `groupKey`
- 把同 `groupKey` 的告警放到 `groupCacheMap` 的 `GroupAlertCache`
- 首次到达 `createTime + groupWait` 发送
- 之后每 `groupInterval` 再次发送
- 若本次全是 `RESOLVED`,立刻发送
- 持续 firing 时按 `repeatInterval` 间隔重复发送(防止长期告警被遗忘)

### 4.3 抑制 `AlertInhibit`

字段: `sourceLabels`、`targetLabels`、`equalLabels`、`enable`(`AlertInhibit.java`)

行为(`AlarmInhibitReduce.java`):
- 对 firing 告警 A:若其 `labels ⊇ sourceLabels`,把 A 缓存为"源告警"
- 对任何告警 B:若其 `labels ⊇ targetLabels` 且在"源告警"中存在 C 满足 `equalLabels` 字段相同 → B 被抑制(从 groupAlert 中移除)
- 源告警 TTL 默认 4h,可通过 `alerter.inhibit.ttl` 配置(`AlerterProperties.java:104-108`)

### 4.4 静默 `AlertSilence`

字段: `name`、`enable`、`matchAll`、`type`(0=单次 1=周期)、`times`(单次生效条数)、`labels`(匹配用)、`days`(周期)、`periodStart/periodEnd`(`AlertSilence.java`)

行为(`AlarmSilenceReduce.java`):
- `matchAll=true` → 全量命中;否则按 labels 任一精确匹配
- 单次(type=0): 命中后 `times--`, 直到 `times=0` 才解除,期间不再下发
- 周期(type=1): 命中 + 当前星期 ∈ `days` + 当前时间在 [periodStart, periodEnd] 区间内 → 静默
- 区间支持跨天(如 23:00~02:00)

### 4.5 通知策略 `NoticeRule` + 接收人 `NoticeReceiver`

通知策略(`NoticeRule.java`):
- 关联 `receiverId[]`(接收人)与 `templateId`(模板)
- `filterAll=true` 命中所有告警;否则按 `labels` 严格匹配 commonLabels
- `days` / `periodStart` / `periodEnd`: 限定生效星期与时间段

接收人(`NoticeReceiver.java`): `type` 决定通道,字段如下:

| type | 名称 | 关键字段 |
|------|------|----------|
| 0 | SMS | `phone` + `SmsClient` 实现(Twilio/Aliyun/AWS/Tencent/UniSms/Local) |
| 1 | Email | `email` + `EmailAlertNotifyHandlerImpl`(可走 DB `general_config` 覆盖 yml) |
| 2 | Webhook | `hookUrl` + `hookAuthType(None/Basic/Bearer)` + `hookAuthToken` |
| 3 | 微信公众号 | `wechatId` |
| 4 | 企业微信机器人 | `accessToken` |
| 5 | 钉钉机器人 | `accessToken` |
| 6 | 飞书机器人 | `accessToken` |
| 7 | Telegram Bot | `tgBotToken` + `tgUserId` + `tgMessageThreadId` |
| 8 | Slack Webhook | `slackWebHookUrl` |
| 9 | Discord Bot | `discordChannelId` + `discordBotToken` |
| 10 | 企业微信 App | `corpId` + `agentId` + `appSecret` + `partyId/tagId` |
| 11 | Slack | `slackWebHookUrl` |
| 12 | Discord | `discordChannelId` + `discordBotToken` |
| 13 | Gotify | `gotifyToken` |
| 14 | 飞书 App | `appId` + `larkReceiveType(0用户/1群/2部门/3全部)` + `chatId` |
| 15 | Server 酱 | `serverChanToken` |
| 16 | 华为云 SMN | `smnAk/smnSk/smnProjectId/smnRegion/smnTopicUrn` |

实现统一继承 `AbstractAlertNotifyHandlerImpl`,流程:
1. `AlertNoticeDispatch.sendNoticeMsg(receiver, template, alert)` 查 `alertNotifyHandlerMap[receiver.type]`
2. `AbstractAlertNotifyHandlerImpl.renderContent()` 用 **Freemarker** 把模板渲染为最终消息(注入 `title/status/groupLabels/commonLabels/commonAnnotations/alerts/consoleUrl`)
3. 各实现类调用对应 SDK(HTTP/JavaMailSender/Java Telegram Bot SDK/华为云 SMN SDK …)

### 4.6 通知执行线程模型

`AlerterWorkerPool`(`alert/AlerterWorkerPool.java`):
- `workerExecutor`(10 线程): 实时指标消费
- `notifyExecutor`(默认 6 线程,虚拟线程模式可配): 实际通知发送,**支持按 channelType 信号量限流**(`executeNotify(byte, Runnable)`)
- `logWorkerExecutor`(默认 10 线程): 日志告警 Reduce 任务

`AlarmCommonReduce` 自身再包一层 Reduce 异步线程池(`alerter-reduce-worker-*`),通过 `reduceAndSendAlarm()` 异步串入 Reduce 链。

### 4.7 落库 & SSE

- 落库: `DbAlertStoreHandlerImpl.store(groupAlert)`(`alert/notice/impl/DbAlertStoreHandlerImpl.java`)
  - 按 fingerprint 增量更新 `hzb_alert_single`(triggerTimes/startAt/endAt 等)
  - 按 groupKey 增量更新 `hzb_alert_group`(合并 commonLabels/Annotations)
- 实时推送: `AlertSseManager.broadcast(json)`(在 `AlertNoticeDispatch.dispatchAlarm` 末尾调用),Web 端通过 `AlertSseController` 订阅

### 4.8 外部告警接入

`ExternAlertService` 实现类: `PrometheusExternAlertService`、`AlertManagerExternAlertService`、`GrafanaExternAlertService`、`SkyWalkingExternAlertService`、`TencentExternAlertService`、`AlibabaCloudSlsExternAlertService`、`HuaweiCloudExternAlertService`、`VolcEngineExternAlertService`、`UptimeKumaExternAlertServiceImpl`、`ZabbixExternAlertServiceImpl`、`DefaultExternAlertService`。
入口: `AlertReportController`(POST `/api/alert/report`)。
外部告警不走规则匹配,直接封装 `SingleAlert` → `AlarmCommonReduce`。

## 6. 一句话总结

HertzBeat 的告警机制 = **AlertDefine(JEXL/ANTLR 表达式 + 阈值 + 模板) 经由 实时消费/周期调度 命中数据 → AlarmCacheManager 去重并累积触发次数 → AlarmGroupReduce/InhibitReduce/SilenceReduce 三层策略过滤 → AlertNoticeDispatch 匹配 NoticeRule + NoticeReceiver + NoticeTemplate → 多通道通知 + 落库 + SSE 推送**。用户配置全部通过 `/api/alert/*` 与 `/api/notice/*` 两套 REST 接口完成。
