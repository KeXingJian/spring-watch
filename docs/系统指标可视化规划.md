# 系统指标可视化规划

## 一、可用指标清单(来自 mock-test `/metrics`)

| 指标名 | 类型 | 单位 | 标签 | 数据特征(示例) |
|---|---|---|---|---|
| `system_disk_io_bytes_total` | counter | bytes | `device`、`direction` | 5 块盘 × 读/写,数量级差异大(0 ~ 1.1E10) |
| `system_disk_operations_total` | counter | ops | `device`、`direction` | 5 块盘 × 读/写,读多写少或反之 |
| `system_memory_usage_bytes` | gauge | bytes | `state` | free/used 二态,适合比例展示 |
| `system_memory_utilization` | gauge | ratio | `state` | free/used(0~1),本质同 bytes,二选一 |
| `system_network_errors_total` | counter | errs | `device`、`direction` | 多数时间为 0,告警语义强 |
| `system_network_io_bytes_total` | counter | bytes | `device`、`direction` | 接收/发送量级对比 |
| `system_network_packets_total` | counter | pkts | `device`、`direction` | 接收/发送包量 |

## 二、当前已渲染 vs 缺失

位置:`src/main/resources/static/pages/app-detail.html → renderOs()`

| 图表 | 当前状态 | 备注 |
|---|---|---|
| 内存时序图(used/free 堆叠面积) | ✅ 已渲染 | `system_memory_usage_bytes`,`#os-mem-chart` |
| CPU 时序图(user/system) | ✅ 已渲染 | `runtime_java_cpu_time_milliseconds`,`#os-cpu-chart` |
| 磁盘 IO 字节(按 device) | ⚠️ 仅聚合 | 5 块盘合并为 1 根柱,丢失 read/write 方向 |
| 网络 IO 字节(按 device) | ⚠️ 仅聚合 | 仅 1 张图,丢失 receive/transmit 方向 |
| 磁盘操作次数(IOPS) | ❌ 缺失 | `system_disk_operations_total` |
| 磁盘 IO 字节(分方向) | ❌ 缺失 | read/write 拆分 |
| 网络错误数 | ❌ 缺失 | `system_network_errors_total` |
| 网络收发包数 | ❌ 缺失 | `system_network_packets_total` |
| 网络 IO 字节(分方向) | ❌ 缺失 | receive/transmit 拆分 |
| 内存使用率仪表盘 | ❌ 缺失 | `system_memory_utilization` 适合做大圆环/进度条 |

## 三、推荐图表方案

### 1. CPU 模块
- **CPU 时序折线图**(已存在):user / system 两条折线
- **新增 CPU 利用率堆叠面积图**:可选 `system_cpu_utilization`,user / system / iowait 三段堆叠

### 2. 内存模块
- **内存时序堆叠面积图**(已存在,used/free)
- **新增 内存使用率仪表盘**:以 `system_memory_utilization{state="used"}` 作大圆环/进度条,百分比直读
- **新增 内存分布饼图**:free / used 两块饼,体积小、读数直观

### 3. 磁盘模块
| 图表 | 类型 | 维度 | 数据来源 |
|---|---|---|---|
| 磁盘 IO 字节(分方向) | **分组柱状图** | X=device,系列=read/write | `system_disk_io_bytes_total`,按 device 分组后内部按 direction 拆分 |
| 磁盘 IOPS(分方向) | **分组柱状图** | X=device,系列=read/write | `system_disk_operations_total`,同上 |
| 磁盘 IO 字节时序 | **堆叠面积图** | 系列=各 device | `system_disk_io_bytes_total` series(取 top N 设备) |
| 磁盘 IOPS 时序 | **折线图** | 系列=各 device | `system_disk_operations_total` series |
| 磁盘使用率(可选) | **水平柱状图** | Y=device,X=used% | `system_filesystem_usage` 类(待确认是否采集) |

### 4. 网络模块
| 图表 | 类型 | 维度 | 数据来源 |
|---|---|---|---|
| 网络 IO 字节(分方向) | **分组柱状图** | X=device,系列=receive/transmit | `system_network_io_bytes_total` |
| 网络包数(分方向) | **分组柱状图** | X=device,系列=receive/transmit | `system_network_packets_total` |
| 网络 IO 字节时序 | **双向折线图(双 Y 轴)** | 上轴=receive,下轴=transmit | `system_network_io_bytes_total` series |
| 网络包数时序 | **堆叠面积图** | 系列=receive/transmit | `system_network_packets_total` series |
| 网络错误数 | **柱状图 + 阈值告警色** | X=device,系列=receive/transmit | `system_network_errors_total`,>0 即高亮红 |

> 计数器(counter)在单点查询时只能拿到"累计值",柱状图展示的是"历史总量";时序图通过 `rate()` 才能展示真实速率。当前后端 `querySeries` 已支持 `agg=rate`,规划时序图统一传 `agg=rate`,柱状图用 `latest` 取最新累计值。

## 四、布局建议(OS 区卡片重组)

```

┌─ 内存区 ────────────────────┐  ┌─ CPU 区 ───────────────────┐
│ 内存时序(堆叠面积)         │  │ CPU 时序(双折线)           │
│ [已渲染 #os-mem-chart]     │  │ [已渲染 #os-cpu-chart]     │
└─────────────────────────────┘  └────────────────────────────┘

┌─ 磁盘 IO 字节(分方向柱) ──┐  ┌─ 磁盘 IOPS(分方向柱) ──────┐

└────────────────────────────┘  └────────────────────────────┘
```

## 五、落地建议

1. **后端**:现有 `/api/metrics/grouped` 只支持单标签分组,新增"按 device + direction 双标签分组"需后端聚合层支持,可在 `MetricQueryService` 增加 `queryMultiGrouped(metric, groupBy[])` 方法,或在前端两次 grouped 后在 JS 端合并。
2. **前端**:以 `renderOs()` 为锚点,先把"分方向柱状图"补齐(实现成本最低);时序图涉及 `agg=rate` 切换,需确认后端已支持并写好测试。
3. **优先级**:磁盘分方向柱状图 > 网络分方向柱状图 > 网络错误告警柱 > 时序图系列(堆叠/双轴)。

## 六、问题与待确认

- `system_filesystem_*`(使用率、剩余空间)是否在采集?示例 `/metrics` 截图中未出现,如未采集则"磁盘使用率"模块暂搁置。
- 后端 `grouped` API 是否支持多维标签组合(先按 device 分,再按 direction 二级分组)? ✅ 已扩展为 `groupBy=device,direction` 逗号分隔

# JVM 指标可视化规划(同 OS 风格)

## 一、可用指标清单

| 指标名 | 类型 | 单位 | 标签 | 示例数据 |
|---|---|---|---|---|
| `jvm_class_count` | gauge | classes | - | 13967 |
| `jvm_class_loaded_total` | counter | classes | - | 14184 |
| `jvm_class_unloaded_total` | counter | classes | - | 217 |
| `jvm_cpu_count` | gauge | cores | - | 12 |
| `jvm_cpu_recent_utilization` | gauge | ratio | - | 7.69E-4 |
| `jvm_cpu_time_seconds_total` | counter | s | - | 195.4 |
| `jvm_gc_duration_seconds` | histogram | s | `jvm_gc_name`、`jvm_gc_action` | G1 Young: 79 次 / 0.51s;G1 Concurrent: 48 次 / 0.65s |
| `jvm_memory_committed_bytes` | gauge | bytes | `jvm_memory_pool_name`、`jvm_memory_type` | 8 个 pool |
| `jvm_memory_limit_bytes` | gauge | bytes | `jvm_memory_pool_name`、`jvm_memory_type` | 5 个 pool 有限制值 |
| `jvm_memory_used_after_last_gc_bytes` | gauge | bytes | `jvm_memory_pool_name`、`jvm_memory_type` | 3 个 pool |
| `jvm_memory_used_bytes` | gauge | bytes | `jvm_memory_pool_name`、`jvm_memory_type` | 8 个 pool |
| `jvm_thread_count` | gauge | threads | `jvm_thread_daemon`、`jvm_thread_state` | 6 个组合 |

## 二、当前已渲染 vs 缺失

位置:`app-detail.html → renderJvm()`

| 图表 | 当前状态 | 备注 |
|---|---|---|
| CPU 使用率(数字卡) | ✅ | `jvm_cpu_recent_utilization` |
| 当前加载类(数字卡) | ✅ | `jvm_class_count` |
| 累计已加载类(数字卡) | ✅ | `jvm_class_loaded_total` |
| 活跃线程(数字卡) | ⚠️ 仅非 daemon + runnable | 漏掉 daemon/总线程 |
| 堆内存使用堆叠时序 | ✅ | `jvm_memory_used_bytes` × heap |
| 非堆内存使用堆叠时序 | ✅ | `jvm_memory_used_bytes` × non_heap |
| GC 累计次数柱 | ⚠️ 仅按 `gc_name` | 漏掉 `gc_action` 维度(共 2 种) |
| 线程状态分布饼 | ⚠️ 仅按 `state` 聚合 | daemon 信息被合并 |
| **GC 耗时(P50/P95/P99)** | ❌ | histogram-quantile API 暂不支持 tag 过滤 |
| **GC 平均/最大耗时** | ❌ | `_sum / _count` 算均值 |
| **类加载/卸载速率** | ❌ | `agg=rate` 看每秒加载数,捕捉类加载抖动 |
| **各 pool 已用占比** | ❌ | `jvm_memory_used / jvm_memory_limit`,定位接近 OOM 的池 |
| **committed 趋势对照** | ❌ | 看 pool 实际能用的字节 |
| **GC 后内存** | ❌ | `_after_last_gc` 反映老年代真实占用 |
| **JVM CPU 时间累计** | ❌ | 数字卡:`jvm_cpu_time_seconds_total` |

## 三、推荐图表方案

### 1. 卡片区
| 卡 | 取数 |
|---|---|
| CPU 使用率(已) | `jvm_cpu_recent_utilization` |
| 当前加载类(已) | `jvm_class_count` |
| 累计已加载类(已) | `jvm_class_loaded_total` |
| **累计已卸载类(新)** | `jvm_class_unloaded_total` |
| 活跃线程(改) | 改为"总线程数",由 `jvm_thread_count` 全部求和 |
| **JVM CPU 时间累计(新)** | `jvm_cpu_time_seconds_total` |

### 2. 内存区
| 图表 | 类型 | 数据来源 |
|---|---|---|
| 堆内存使用堆叠(已) | 时序堆叠面积 | `jvm_memory_used_bytes` × heap |
| 非堆内存使用堆叠(已) | 时序堆叠面积 | `jvm_memory_used_bytes` × non_heap |
| **各 pool 已用 vs 上限** | 横向条形,used% 着色 | `jvm_memory_used_bytes / jvm_memory_limit_bytes` 单点 |
| **堆 committed 趋势** | 时序线,各 pool 拆分 | `jvm_memory_committed_bytes` × heap |
| **GC 后剩余内存** | 柱图(各 pool) | `jvm_memory_used_after_last_gc_bytes` |

### 3. GC 区
| 图表 | 类型 | 数据来源 |
|---|---|---|
| **GC 累计次数(替换)** | 分组柱 `name × action` | `jvm_gc_duration_seconds_count` grouped by `jvm_gc_name,jvm_gc_action` |
| **GC 平均耗时** | 柱图,按 `gc_name` | `_sum / _count` 算 mean,按 name 聚合 |
| **GC P50/P95/P99** | 多线时序图,按 `gc_name` | histogram-quantile,需 API 支持 tag 过滤 |
| **GC P99 大值告警** | 在 P99 折线图上对 > 阈值(如 200ms)上红 | 同上 |

### 4. 线程/类加载 区
| 图表 | 类型 | 数据来源 |
|---|---|---|
| **线程 daemon × state 双维** | 分组柱,daemon=true/false × state | `jvm_thread_count` grouped by `jvm_thread_daemon,jvm_thread_state` |
| **类加载速率** | 时序线,`agg=rate` | `jvm_class_loaded_total` |
| **类卸载速率** | 时序线,`agg=rate` | `jvm_class_unloaded_total` |

## 四、布局建议(替换现有 4 卡 + 2 行)

```
┌─ 顶部 6 个数字卡 ────────────────────────────────────────────────┐
│ CPU 使用率 │ 当前加载类 │ 累计已加载类 │ 累计已卸载类 │           │
│ 总线程数   │ JVM CPU 时间累计(秒)                               │
└──────────────────────────────────────────────────────────────────┘

┌─ 内存区(2 行)───────────────────────────────────────────────────┐
│ 行1: 堆内存使用堆叠(已有)  │ 非堆内存使用堆叠(已有)             │

┌─ GC 区(2 行)────────────────────────────────────────────────────┐
│ 行1: GC 累计次数(name×action)  │ GC 平均耗时(name)              │

┌─ 线程/类加载 区 ─────────────────────────────────────────────────┐

└──────────────────────────────────────────────────────────────────┘
```

## 五、落地优先级

1. **后端**:扩展 `histogram-quantile` API 支持 tag 过滤(必做,GC P95/P99 多线依赖)
2. **前端**:
   - P0:GC 累计次数(name×action)、GC 平均耗时、类加载/卸载速率、线程 daemon×state、内存各 pool 占比
   - P1:GC P50/P95/P99 多线、堆 committed 趋势、GC 后内存
   - P2:数字卡扩展(累计已卸载类、JVM CPU 时间累计、总线程数)

## 六、待确认

- `jvm_memory_limit_bytes` 在示例中只有 5 个 pool 有值(G1 Eden、Survivor 无 limit 字段,因 G1 不预先划分)——占比图需在缺失 limit 的 pool 上做"无上限"提示
- `jvm_class_loaded_total` 是 counter,速率图需用 `agg=rate` 拉取,InfluxDB 该指标若已写过 rate 是否会被重置?需测试确认
