## 二、队列与 OOM 风险矩阵

### 2.1 DelayedWorkQueue 详解(11 处共用此结构)

**结构**:`ScheduledThreadPoolExecutor` 内部专用,数组+堆实现的优先队列,按 trigger time 排序,**无 size 限制**。

**实际行为分析**:

```
class DelayedWorkQueue {
  add(task):
    1. 数组扩容检查(若超容量,数组翻倍,无固定上限)
    2. siftUp 到正确位置
}
```

**是否会 OOM?取决于业务怎么用**:

| 用法 | 风险 |
|------|------|
| `scheduleWithFixedDelay` (你用这个) | **不会累积**。每次 fire 后 re-schedule 同一 future,新 trigger time 替换旧 trigger time。**队列大小 = 注册任务数 = 固定** |
| `scheduleAtFixedRate` + 任务慢于周期 | **会累积**!漏掉的周期会堆进队列,极端可 OOM |
| `schedule` 一次性 + 手动重排 (你用 cronLoop) | **不会累积**。`jobs.put` 覆盖老 future,老 entry 失去引用可 GC |
| 循环中反复 `schedule` 不取消 | **会累积**(高危) |

**你项目里 11 个 scheduler 全部属于前 3 类安全用法** ✓

### 2.2 各队列风险评分

| 队列 | 类型 | 上限 | 行为 | OOM 风险 | 监控 |
|------|------|------|------|----------|------|
| `DelayedWorkQueue × 11` | 优先 | 实际=任务数 | `scheduleWithFixedDelay` 不累积 | 🟢 极低 | ❌ 无 |
| `PullRetryQueue.queue` | 优先 | `maxQueueSize=1000` 软限 | 满 → 拒(rejectedCounter) | 🟢 低 | ✅ `retry.queue.size` |
| `KafkaFallbackQueue.queue` | 阻塞 | `capacity=10000` 硬限 | 满 → 丢 | 🟢 低 | ✅ `kafka.fallback.queue.size` |
| `PendingStateScanner` `LinkedBlockingQueue` | 阻塞 | **无界** ⚠️ | 每 5s `execute(this::doScan)`,老任务未完成就堆 | 🟡 中 | ❌ 无 |
| `AsyncAlertExecutor` 任务 | 信号量 | `poolSize=8` | 满 → 同步降级 | 🟢 低 | ❌ 无 |
| `AsyncMailExecutor` 任务 | 信号量 | `poolSize=4` | 满 → 同步降级 | 🟢 低 | ❌ 无 |
| `AgentHttpClient` 任务 | 信号量 | per-host 80 | 满 → 快速失败 | 🟢 低 | ✅ `http.active` |
| Tomcat 线程池 | 阻塞 | 200 | 满 → 排队或拒 | 🟢 低 | Spring 默认 |
| ForkJoinPool.common | 无界 | 32767 | `parallelStream` | 🟢 低 | JVM 默认 |
| Kafka listener 内部 | Kafka 管 | Kafka 管 | Kafka 管 | 🟢 极低 | Kafka JMX |
| `@Scheduled` `DelayedQueue` | 优先 | 任务数(5) | 5 个 cron 任务 | 🟢 极低 | Spring 默认 |

## 三、3 个需要关注的点

### 🔴 关注 1:`PendingStateScanner` 用了无界 `LinkedBlockingQueue`

```java
// PendingStateScanner.java:40
this.scanExecutor = Executors.newSingleThreadExecutor(tf);
```

**问题**:
- `newSingleThreadExecutor` 内部用 `LinkedBlockingQueue` (无界,Integer.MAX_VALUE)
- `@Scheduled` 5s 触发一次,如果 `doScan` 执行 > 5s,任务会无限堆在队列
- 1000 个 app 规则扫描慢的话,几个小时内就堆到几十万 entry
- 每个 entry 是 `Runnable` 引用 ≈ 32 字节,几十万才几 MB,真正风险是 GC 压力 + 内存泄漏(老任务持有规则状态)

**修复**(建议):
```java
// 方案 A:加有界队列 + 拒绝策略
ThreadFactory tf = Thread.ofVirtual().name("pending-scanner-", 0).factory();
this.scanExecutor = new ThreadPoolExecutor(
        1, 1, 0L, TimeUnit.MILLISECONDS,
        new LinkedBlockingQueue<>(8),  // 只留少量缓冲
        tf,
        new ThreadPoolExecutor.AbortPolicy());
```

**或 方案 B:加 `fixedRate` 改为 `fixedDelay`** (Spring `@Scheduled` 已用 `fixedDelay`,只需确保不重入):
```java
// 已经是 fixedDelay,问题在 executor 重入
// 方案:execute 之前检查 executor 是否忙碌
```

### 🟡 关注 2:`AsyncAlertExecutor` / `AsyncMailExecutor` Semaphore 满时同步降级

```java
// AsyncAlertExecutor.java:65
try {
    semaphore.acquire();
} catch (InterruptedException ie) { ... }
try {
    executor.execute(() -> { ... });
} catch (Exception e) {
    semaphore.release();
    log.warn("提交失败, 降级为同步");
    task.run();  // ← 同步执行!
}
```

**问题**:
- Semaphore 满 → 同步执行 → 调用方线程(Kafka consumer)被 SMTP/规则评估阻塞
- 突发告警时,所有 Kafka consumer 线程被拖住,Kafka 消费暂停
- `execute()` 本身不会失败(虚拟线程 per-task 不会拒绝),但中断会触发

**建议**:失败时不"降级同步",而是丢入一个有界 retry queue 或直接丢弃,确保消费者不阻塞。

### 🟡 关注 3:11 个 `DelayedWorkQueue` 完全没有监控

**问题**:虽然分析上不会 OOM,但如果:
- 未来有人改用 `scheduleAtFixedRate`
- 有人忘记 `cancelInternal` 就 add new
- cronLoop 出 bug 没重排

**不会立刻被发现**(没监控,只能看 GC)。

**建议**:`SelfMonitorCollector` 加一个统一 gauge,反射读所有 scheduler 的 queue size,聚合暴露。

## 五、最终结论

| 项 | 数值 |
|----|------|
| 自建 Executor/Scheduler | 16 处 |
| 其中用无界队列 | 1 处(`PendingStateScanner`) |
| 真实 OOM 风险 | 1 处需修,0 处会爆 |
| 监控覆盖 | 3/16 队列有 gauge,13/16 缺监控 |
| 总体健康度 | 🟡 中等(主流程稳,边角盲点) |

**主线程模型设计良好**(全用虚拟线程 + Semaphore 两级限流 + 有界降级队列),**只 `PendingStateScanner` 这一处是历史遗留 bug**,其余都是分析可证明安全的。
