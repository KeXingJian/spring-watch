## 〇、TL;DR

| 维度 | 当前 spring-watch | HertzBeat | 借鉴动作 |
|---|---|---|---|
| HTTP 客户端 | JDK `HttpClient` + `BodyHandlers.ofByteArray()`(整 body 入堆) | Apache `PoolingHttpClientConnectionManager`(maxTotal=50000 / perRoute=80) + 流式 `entity.getContent()` | **P0**:换池化客户端 + InputStream 直传 |
| Prometheus 解析 | `body.lines() + split` 二次解码 | `OnlineParser` 字符级状态机(读一个字符解析一个字符) | **P0**:复制 OnlineParser |
| 单次拉取往返数 | 探测 1 + 拉 metrics 1 + 拉 logs 1 = **3 RTT** | 单次 HTTP 调用 + 一次 INLINE 流式解析 | **P0**:探测与首请求复用同连接 / 探测与拉取合一 |
| 限流后行为 | 入 `PullRetryQueue`,drainer **重走整套 doHeavyWork**(含可达性探测) | `LIMIT_AND_REJECT` + 降优先级原地重试 | **P1**:重投只重试 `heavyWork`,跳过探测与心跳 |
| Kafka 写入 | 每条 `sendMetric`/`sendLog` 一次 `kafkaTemplate.send` | 解析完后走队列 → 批写 | **P0**:指标/日志本地攒批,按 N 条或 T ms 刷写 |
| 调度器 | `ScheduledThreadPool` + `scheduleWithFixedDelay`(线程栈占用) | `HashedWheelTimer`(1s tick / 512 槽) | **P2**:CRON 与 INTERVAL 切到时间轮 |
| 任务准入 | `HostThrottler` Semaphore(per-host=4),无全局限 | `ManagedExecutors` 三模式(UNBOUNDED_VT / LIMIT_AND_REJECT / LIMIT_AND_BLOCK)+ `DiscardOldestVirtualManagedExecutor` | **P1**:三模式 + 丢最老降级 |
| 长连接 | 无(每次 `HttpRequest` 重用连接由 JDK 池决定) | Netty `IdleStateHandler(100s)` + `Epoll` 自动切换 | **P3(可选)**:Agent 改造为 Netty Server,spring-watch Netty Client 长连 + 半双工推送 |
| 调度超时监控 | 无 | `CommonDispatcher.monitorCollectTaskTimeout` 每 20s 巡检,> 4min 强制丢弃 | **P1**:拉取超 30s 直接 cancel |

> **最值得借鉴的 3 点(按 ROI)**:
> 1. **HertzBeat 的 `parseResponseByPrometheusExporter(entity.getContent(), ...)`**:边读 socket 边解析,避免"body→byte[]→String→split"三连转换
> 2. **HertzBeat 的 `CommonHttpClient` 池化配置 + 后台 evict**:maxPerRoute=80,远高于 JDK 默认
> 3. **HertzBeat 的"限流 + 降优先级"原地重试**:拒绝 = 退避 + runPriority+1 重投,而不是重走整条链路

## 一、问题定位:当前 spring-watch 上游拉取链路的 5 个具体瓶颈

### 1.1 链路全景

```
CollectScheduleRegistry(32 线程 ScheduledExecutor,虚拟线程)
    └─> safeRun(appid)
        └─> AppPullTask.run(appid)                [同步串行]
            ├─ ① monitorAppRepository.findByAppid    [DB 1 次]
            ├─ ② isReachable → agentHttpClient.get   [HTTP 1 RTT,探测 /metrics]
            ├─ ③ sendHeartbeat → kafka.send          [Kafka 1 send]
            ├─ ④ hostThrottler.tryAcquire            [Semaphore 准入]
            ├─ ⑤ doHeavyWork(appid)
            │     ├─ ⑤-1 monitorAppRepository.findByAppid  [DB 1 次,与 ① 重复]
            │     ├─ ⑤-2 agentMetricsCollector.collect
            │     │     └─ agentHttpClient.get(/metrics)   [HTTP 1 RTT,同步]
            │     │        └─ byte[] body → UTF-8 decode   [全 body 入堆]
            │     │        └─ body.lines() 逐行解析          [二次切分]
            │     │        └─ forEach sendMetric → kafka    [N 次 send,N=指标行数]
            │     └─ ⑤-3 agentLogCollector.collect(since)
            │           └─ agentHttpClient.get(/api/agent/logs?since=)  [HTTP 1 RTT,同步]
            │              └─ byte[] body → JSON array 解析           [JsonParser 流式,OK]
            │              └─ forEach sendLog → kafka                  [N 次 send,N=日志条数]
            └─ ⑥ hostThrottler.release
```

**1 次拉取周期 = 2 DB + 2~3 HTTP + 1+N+M Kafka send**。

### 1.2 瓶颈 ①:`BodyHandlers.ofByteArray()` 全量堆分配

**位置**:`collector/AgentHttpClient.java:91`

```java
HttpResponse<byte[]> resp = httpClient.send(req, HttpResponse.BodyHandlers.ofByteArray());
...
byte[] body = resp.body();
if (body != null && body.length > maxBodyBytes) {  // maxBodyBytes=4MB
    bodyTooLargeCounter.increment();
    ...
}
String text = body == null ? "" : new String(body, StandardCharsets.UTF_8);
```

**问题**:
- 每次拉取 `/metrics`(一个中等规模 Java Agent 通常 30~100KB,prometheus 文本+ tags)→ 在堆上同时存在 `byte[100K]` + `String(100K)`,GC 压力随 app 数量线性增长
- `maxBodyBytes=4MB` 是一颗定时炸弹:单个错乱 Agent 返回 4MB+ 时,整个服务 Young GC 至少 1 次
- 拉取与解析是**串行**:必须等 body 全部收完才开始解析

**压测可观察指标**:`spring.watch.collector.http.active` 高水位、`jvm.gc.pause` 频繁、Young Gen 占用 > 60%

### 1.3 瓶颈 ②:`body.lines() + lastIndexOf(' ')` Prometheus 解析低效

**位置**:`collector/AgentMetricsCollector.java:42-51`

```java
body.lines()
    .filter(line -> !line.isBlank() && !line.startsWith("#"))
    .map(this::parsePrometheusLine)
    ...
```

- `body.lines()` 返回 `Stream<String>`,每行调一次 `String.split` 产生中间 `String[]`
- `parsePrometheusLine` 内部 `lastIndexOf(' ')` + `substring` 多次产生中间字符串
- 单次 `appid` 拉 1 次 `/metrics` 产生 **N 行 × 3~5 个中间 String**
- 一个 100 应用的集群,15s 周期下每秒产生 600+ 中间 String 对象

**对比 HertzBeat**:`prometheus/parser/OnlineParser.java:62-85` 直接 `InputStream` 按字符读取:

```java
int i = getChar(inputStream);
while (i != -1) {
    if (i == '#' || i == '\n') { skipToLineEnd(inputStream).maybeEol()... }
    else { ... parseMetric(inputStream, ...) }
    i = getChar(inputStream);
    if (i == '\r') i = getChar(inputStream);
}
```

- 全程 1 个 `StringBuilder` 复用,**零中间 String**
- 边读 socket 边解析,内存峰值 = 单行 Prometheus 文本

### 1.4 瓶颈 ③:1 个拉取周期 = 3 RTT(可达性 + metrics + logs)

**位置**:`collector/AppPullTask.java:48-107`

- `isReachable()` 单独发 1 次 GET `/metrics`,仅返回 200/非 200
- 紧接着 `agentMetricsCollector.collect()` 又发 1 次 GET `/metrics`(同一 URL)
- 然后 `agentLogCollector.collect()` 再发 1 次 GET `/api/agent/logs`

**问题**:
- 探测与首请求**走两条独立 HTTP 路径**,即使 Agent 健康,3 RTT 中前 2 RTT 是冗余
- `connect-timeout-ms:3000` + `read-timeout-ms:10000`,Agent 卡顿时探测本身就要 3s,后续 metrics 请求再 10s,单次 worst case 13s
- `metricsPort(9464)` 与 `endpoint` 是不同 host:port 时,JDK HttpClient 连接池无法跨域复用

### 1.5 瓶颈 ④:被限流后入 `PullRetryQueue`,drainer **重走整套 doHeavyWork**

**位置**:`collector/schedule/PullRetryQueue.java:151-187`

```java
private void processPull(RetryPull pull, String drainerName) {
    MonitorApp app = repository.findByAppid(pull.appid()).orElse(null);  // 重复查 DB
    ...
    if (!throttler.tryAcquire(host, 0)) {  // 重复限流判断
        ...
        scheduler.schedule(() -> enqueue(next), backoff, ...);
        return;
    }
    try {
        appPullTask.doHeavyWork(pull.appid());  // 实际只重做拉取
    } finally {
        throttler.release(host);
    }
}
```

- drainer 拿到 `RetryPull` 后,先查 DB、再判限流、再调 `doHeavyWork`
- 与 `AppPullTask.run` 的前半段(DB 查 + 限流)重复,但代码路径分开维护
- 退避采用 `500ms << attempt`(attempt 5 即 16s 退避),期间 app 状态可能在 DB 改了

**对比 HertzBeat**:`CommonDispatcher.start()`

```java
workerPool.executeLongRunning(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        MetricsCollect metricsCollect = jobRequestQueue.getJob();
        if (metricsCollect != null) {
            try { workerPool.executeJob(metricsCollect); }
            catch (RejectedExecutionException rejected) {
                // 降优先级 + 原地重投, 不查 DB 不重判限流
                metricsCollect.setRunPriority((byte)(metricsCollect.getRunPriority() + 1));
                jobRequestQueue.addJob(metricsCollect);
            }
        }
    }
});
```

- 拒绝时只**改 runPriority +1 + 重新入队**,不重新构造任务、不查 DB
- `PriorityBlockingQueue` + `Comparable` 自动按优先级出队
- drainer 永远只有 1 个循环,无 drainerPool 维护成本

### 1.6 瓶颈 ⑤:`scheduleWithFixedDelay` 32 线程 + 每个 app 1 个 Future

**位置**:`collector/schedule/CollectScheduleRegistry.java:40-44`、`:105-109`

```java
this.scheduler = Executors.newScheduledThreadPool(properties.getPoolSize(), tf);  // poolSize=32
...
future = scheduler.scheduleWithFixedDelay(
        () -> safeRun(app.getAppid()),
        initialDelay,
        schedule.baseIntervalMs(),  // 15s
        TimeUnit.MILLISECONDS);
```

- 200 个被监控应用 → 200 个 `ScheduledFuture`
- 32 线程虚拟线程池拉 200 个任务,每 15s 一波,峰值并发 = `min(32, 200) = 32`
- 应用规模 > 200 时,延迟开始堆积(`scheduleWithFixedDelay` 模式)
- `ScheduledThreadPool` 内部用 `DelayedWorkQueue`(最小堆),O(log n) 调度

**对比 HertzBeat**:`TimerDispatcher` 使用 `HashedWheelTimer(r -> daemon, 1, TimeUnit.SECONDS, 512)`:
- 1s tick × 512 槽 = ~512s 范围
- 插入/取消 O(1),无堆维护
- 单线程 worker 推进 tick,虚拟线程执行 task
- 万级任务也 O(1)

**改造方案**:

```java
    public NettyCollectorClient(MeterRegistry registry,
                                int maxConnectionsPerHost,         // 默认 80
```

**建议路径**:**保留 JDK HttpClient + 加 1 层 per-host 计数 Semaphore**(等价于 `MAX_PER_ROUTE_CONNECTIONS = 80`),避免引入 Apache HttpClient 5 的额外依赖。

**伪代码(per-host 计数)**:
```java
private final ConcurrentHashMap<String, Semaphore> hostConnPermits = new ConcurrentHashMap<>();
private final int maxPerHost = 80;

public Result get(String url) {
    String host = URI.create(url).getHost();
    Semaphore sem = hostConnPermits.computeIfAbsent(host, _ -> new Semaphore(maxPerHost));
    if (!sem.tryAcquire(connectTimeoutMs, MILLISECONDS)) { /* fail-fast */ }
    try { return doGet(url); } finally { sem.release(); }
}
```

**收益估算**:
- per-host 80 连接 → 单 Agent 突发可同时 80 拉取,不被 JDK 默认 32 限制
- 配合后端 `evictIdleConnections(100s)` → 空闲连接不堆积

### 2.2 🔴 P0-2:`OnlineParser` 字符级流式解析(直接抄)

**借鉴源**:`hertzbeat-collector-basic/.../prometheus/parser/OnlineParser.java:62-193`

**HertzBeat 核心结构**:
```java
public class OnlineParser {
    private static Map<Integer, Integer> escapeMap = new HashMap<>(8);

    public static Map<String, MetricFamily> parseMetrics(InputStream inputStream) throws IOException {
        Map<String, MetricFamily> map = new ConcurrentHashMap<>(10);
        int i = getChar(inputStream);
        while (i != -1) {
            if (i == '#' || i == '\n') skipToLineEnd(inputStream);
            else {
                StringBuilder sb = new StringBuilder();
                sb.append((char) i);
                parseMetric(inputStream, map, sb);  // 字符级状态机
            }
            i = getChar(inputStream);
            if (i == '\r') i = getChar(inputStream);
        }
        return map;
    }

    private static int getChar(InputStream in) throws IOException {
        return in.read();  // ← 单字符读取
    }
}
```

- 1 个 `StringBuilder` 复用

- 边读边解析,**不缓冲整行**(一行解析完即清理 StringBuilder)
- 全程无 `String.split`、无 `String.substring`

**spring-watch 改造**:

### 2.4 🟡 P1-1:Kafka 写入攒批(指标/日志本地 N 条/T ms 刷)

**借鉴源**:HertzBeat 的 `collectDataQueue.sendMetricsData(metricsData)` + NettyDataQueue(`hertzbeat-collector-common/.../dispatch/export/NettyDataQueue.java`)

**spring-watch 当前**(`AgentMetricsCollector.java:48-51`):
```java
.forEach(event -> {
    kafkaProducerBridge.sendMetric(event);  // 1 条事件 = 1 次 kafka.send
    metricCount[0]++;
});
```

- 100 个应用 × 平均 200 指标行/次 × (1/15)Hz ≈ **1333 次 Kafka send/s**
- 每次 `kafkaTemplate.send` 内部 `RecordMetadata` + `Future` 分配,Producer 池(Linger=0?)实际未合并

**改造方案**:在 `AgentMetricsCollector` 出口加 1 层 **本地攒批**:
```java
// 新增:MetricBatcher.java
public class MetricBatcher {
    private final ArrayList<MetricEvent> buffer = new ArrayList<>(2048);
    private final int maxBatchSize = 500;       // N 条刷
    private final long maxBatchDelayMs = 200;   // T ms 刷
    private final KafkaProducerBridge kafka;

    public void offer(MetricEvent e) {
        synchronized (buffer) {
            buffer.add(e);
            if (buffer.size() >= maxBatchSize) flush();
        }
    }

    @Scheduled(fixedDelay = 200)
    public void flushOnTime() {
        synchronized (buffer) {
            if (!buffer.isEmpty()) flush();
        }
    }

    private void flush() {
        List<MetricEvent> batch = new ArrayList<>(buffer);
        buffer.clear();
        batch.forEach(kafka::sendMetric);
    }
}
```

**注**:`spring-kafka` 内部已做 `linger.ms` 合并,**如配置了 `linger.ms=50` 则攒批不必要**;但 spring-watch 当前未配 linger,需先查 `application.yml`。

**行动**:先查 `spring.kafka.producer.properties.linger.ms`,如有 50ms 则跳过此步;否则补 `linger.ms=50, batch.size=65536, compression.type=lz4`。

**收益估算**:
- 1333 send/s → 5~10 批/s,Kafka broker 处理压力降 100x
- Producer 端 `RecordMetadata` 分配降 100x

### 2.5 🟡 P1-2:限流拒绝 = 降优先级原地重投(抄 CommonDispatcher)

**借鉴源**:`hertzbeat-collector-collector/.../dispatch/CommonDispatcher.java:117-140`、`MetricsCollect.java:154`

**HertzBeat 模式**:
```java

}

// CommonDispatcher 拒绝时
catch (RejectedExecutionException rejected) {
    metricsCollect.setRunPriority((byte)(metricsCollect.getRunPriority() + 1));
```

**spring-watch 改造**:
```java
// 改造 PullRetryQueue.processPull 路径
// 改为:被限流时,直接在 MetricsCollect / PullJob 上调 runPriority+1 + offer 回去
public void onThrottled(Long appid, String host) {
    PullJob job = jobRegistry.computeIfAbsent(appid, k -> new PullJob(appid, host));
    job.runPriority += 1;
    jobQueue.offer(job);  // 优先级队列自动按降序出

```

**收益**:
- 重投不重查 DB、不重判限流

### 4.1 `OnlinePrometheusParser` 精简版(spring-watch 专用)

```java
package com.springwatch.collector.parse;

import java.io.IOException;
import java.io.InputStream;
import java.util.HashMap;
import java.util.Map;

public class OnlinePrometheusParser {

    private static final Map<Integer, Integer> ESCAPE_MAP = new HashMap<>(8);
    static {
        ESCAPE_MAP.put((int) '\\', (int) '\\');
        ESCAPE_MAP.put((int) '\"', (int) '\"');
        ESCAPE_MAP.put((int) 'n', (int) '\n');
    }

    public interface MetricLineHandler {
        void onMetric(String name, Map<String, String> tags, double value);
    }

    public static void parse(InputStream in, MetricLineHandler handler) throws IOException {
        int c = in.read();
        StringBuilder sb = new StringBuilder(128);
        while (c != -1) {
            if (c == '#' || c == '\n') {
                skipToEol(in);
                c = in.read();
                if (c == '\r') c = in.read();
                continue;
            }
            sb.setLength(0);
            sb.append((char) c);
            parseOneLine(in, sb, handler);
            c = in.read();
            if (c == '\r') c = in.read();
        }
    }

    private static void parseOneLine(InputStream in, StringBuilder sb, MetricLineHandler h) throws IOException {
        // 1) metric name
        int c = in.read();
        while (c != ' ' && c != '{' && c != -1) {
            sb.append((char) c);
            c = in.read();
        }
        String name = sb.toString();
        sb.setLength(0);

        // 2) tags
        Map<String, String> tags = new HashMap<>(4);
        if (c == '{') {
            c = in.read();
            StringBuilder key = new StringBuilder(32);
            StringBuilder val = new StringBuilder(64);
            boolean inKey = true;
            while (c != '}') {
                if (c == '=') { inKey = false; c = in.read(); continue; }
                if (c == '"') {  // 字符串值
                    c = in.read();
                    while (c != '"') {
                        if (c == '\\') { c = in.read(); val.append((char) ESCAPE_MAP.getOrDefault((int) c, c)); }
                        else val.append((char) c);
                        c = in.read();
                    }
                    c = in.read();  // skip closing "
                    continue;
                }
                if (c == ',') {
                    tags.put(key.toString().trim(), val.toString().trim());
                    key.setLength(0); val.setLength(0); inKey = true;
                    c = in.read(); continue;
                }
                if (inKey) key.append((char) c); else val.append((char) c);
                c = in.read();
            }
            if (key.length() > 0) tags.put(key.toString().trim(), val.toString().trim());
            c = in.read();  // skip '}'
        }

        // 3) skip space
        while (c == ' ') c = in.read();

        // 4) value
        sb.setLength(0);
        while (c != ' ' && c != '\n' && c != -1) {
            sb.append((char) c);
            c = in.read();
        }
        double value = Double.parseDouble(sb.toString());

        h.onMetric(name, tags, value);
    }

    private static void skipToEol(InputStream in) throws IOException {
        int c = in.read();
        while (c != '\n' && c != -1) c = in.read();
    }
}
```

## 五、借鉴边界(**不**建议照搬的部分)

| HertzBeat 设计 | 不建议照搬的原因 |
|---|---|
| 自研 Netty Remoting(Probuf + Zlib 压缩 + IdleStateHandler) | spring-watch 用 Kafka 已满足双向通信,自研 RPC 维护成本 ≥ 5 人天,且需要 Agent 端同步改造 |
| `HashedWheelTimer` 单线程 worker | spring-watch 当前 200 app 以下无需,过早优化 |
| `CommonHttpClient` 静态单例(全局共享) | spring-watch 多实例部署时,静态单例会阻碍依赖注入与单元测试 |
| 整 `consumeQuietly(entity)` 后再 EntityUtils.toString | 这是 HertzBeat 的 fallback 路径,Prometheus 路径已用 InputStream(应抄) |
| `dispatchChannelHealthCheck 3s` 通道巡检 | spring-watch 拉 HTTP 端点而非长连接,1h 周期足够 |
| 优先级 = `(byte) ±1` 极小值域 | spring-watch 建议扩到 0~127(byte 范围)以支持更细粒度 |

## 六、总结

**HertzBeat 的核心价值(对"上游拉取性能"场景)在三处**:
1. **HTTP body 不入堆**:`entity.getContent()` → `OnlineParser` 字符级状态机
2. **连接池化 + 后台清理**:`maxPerRoute=80` + `evictIdleConnections(100s)` + `evictExpiredConnections`
3. **拒绝 = 降优先级原地重投**:拒绝时只改 `runPriority+1` + `offer` 回去,不重建任务

**spring-watch 本期应做的 5 件事(P0)**:
- [ ] 复制 `OnlineParser` 为 `OnlinePrometheusParser`(`BodyHandlers.ofByteArray` → `ofInputStream`)
- [ ] `AgentHttpClient` 加 per-host Semaphore(默认 80)
- [ ] 可达性探测改 HTTP HEAD
- [ ] Kafka producer 加 `linger.ms=50, batch.size=65536, lz4`
- [ ] `Prometheus` 端点暴露(`/actuator/prometheus`),把 5 个新指标(`pull.timeout` / `http.conn_pool_exhausted` 等)送出去

**预期效果(单实例 200 app / 15s 周期)**:
- 单次 `/metrics` 拉取 GC 分配: 100K → 几 KB(-95%)
- 单 Agent 并发拉取上限: 32 → 80
- 单次拉取 RTT: 3 → 2
- Kafka send/s: 1333 → 5~10

> **下一步**:把 P0-1~P0-5 拆 issue,优先实施 P0-1(OnlineParser 复制),可在 0.5 人天内完成,直接收益最显著。
