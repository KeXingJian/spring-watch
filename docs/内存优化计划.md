# spring-watch 内存优化计划

> 本文档为 spring-watch 平台的内存优化专项规划,适用于 v1.2+。
> 优化以**最小实现、聚焦热路径**为原则,不引入与白皮书硬约束冲突的设计变更。
> 参考基线:Java 25 + Spring Boot 4.0.1 + 虚拟线程 + JDK HttpClient + Kafka + InfluxDB v2 + PostgreSQL + Redis。

---

## 1. 现状评估(优化前内存画像)

### 1.1 假设规模(用于估算)

| 维度 | 假设值 | 说明 |
|---|---|---|
| **监控应用数** | 1,000 | `monitor_app` 行数 |
| **采集周期** | 15s/次 | INTERVAL 默认值 |
| **单应用 scrape 体量** | ~10KB / ~200 metric 行 | 中等 OTel Agent |
| **日志峰值吞吐** | 10K events/s 全平台 | 100 app × 100/s |
| **告警规则数** | 50~200 条 | 实际取决于运营 |
| **JVM 堆** | 1~2 GB(默认) | 当前 `application.yml` 未显式设置 |
| **GC** | G1(默认) | 无显式调优 |

### 1.2 内存占用估算(当前)

| 组件 | 单实例体积 | 数量级 | 备注 |
|---|---|---|---|
| `KafkaFallbackQueue` 满载 | ~250 MB | 50K × ~5KB | 极端情况,持续阻塞 Kafka 时积累 |
| InfluxDB WriteApi 缓冲 | ~20 MB | 100K points × ~200B | 写 InfluxDB 慢时积累 |
| Kafka Producer 缓冲 | 128 MB | `buffer-memory: 128MB` | 配置硬上限 |
| `SelfMonitorCollector` 环形 | ~50 MB | 360 × 全量 meter | 每 sample 复制整张 meter 表 |
| `AgentHttpClient` 响应体 | ~5 MB 并发 | 多 body 同时在堆 | 32 线程 × 4 per-host |
| `AlertRuleCache` | ~1 MB | 200 规则 × 5 字段 | 可控 |
| `LogDedupService` dirty set | 无上限 | 持续积压 | 风险点 |
| `HostThrottler` Semaphore Map | ~1 MB | 1000 host × 80B | 可控 |
| `CollectScheduleRegistry` jobs | ~1 MB | 1000 × 100B | 可控 |
| `AlertStateStore.scanPending` | ~10 MB / 扫描 | PENDING × 全字段 | 每次扫加载完整 hash |

**最坏情况合计:约 470 MB** (其中 `KafkaFallbackQueue` 与 `SelfMonitorCollector` 是大头)
**典型情况(无 Kafka 阻塞、InfluxDB 正常):约 200 MB** → JVM 1GB 即可,但缺少 30%~50% 余量应对突发。

## 2. 内存热点分级

> 按 **(占用 × 风险) ÷ 收益** 排序。P0=必做,P1=应做,P2=可做。

### P0 必做(高占用 / 高风险 / 低改动)

|---|---|---|---|---|---|
| P0-1 | Kafka 兜底队列无降载策略 | `KafkaFallbackQueue` | 50K × 5KB ≈ 250MB | Kafka 长时间故障时 OOM | 小 |
| P0-2 | SelfMonitor 每 10s 复制全量 meter | `SelfMonitorCollector.sample` | ~50MB ring | 长跑后 GC 频繁 | 中 |
| P0-3 | Log dedup 每次 4 次 Redis RTT | `LogDedupService.shouldKeep` | 网络 + 临时对象 | 10K log/s 时打爆 Redis | 小 |
| P0-4 | HTTP 响应全量加载到 String | `AgentHttpClient.get` | 单次 10KB~1MB | 大端点 32 线程并发撑爆堆 | 中 |
| P0-5 | Prometheus 解析每行 new HashMap | `AgentMetricsCollector.parsePrometheusLine` | 200 行 × HashMap | 高分配率,Young GC 频繁 | 小 |

### P1 应做(中等收益)

| # | 热点 | 位置 | 占用 | 风险 | 改动量 |
|---|---|---|---|---|---|
| P1-1 | MonitorApp 每次拉取都查 DB | `AppPullTask.run/doHeavyWork` | 67 DB hit/s | 1000 app × 15s 长期 DB 压力 | 小 |
| P1-2 | LogSanitizer 5 次 regex 链式调用 | `LogSanitizer.mask` | 每条 5 个 Pattern 匹配 | 高分配率 | 小 |

| P1-5 | AlertStateStore.scanPending 每次 entries() 加载完整 hash | `AlertStateStore` | 10K PENDING × 5 字段 | 5s 一次,持续内存抖动 | 中 |

## 3. 详细优化方案

### P0-1 Kafka 兜底队列降载

**问题**:`KafkaFallbackQueue` 默认 `capacity=50000` 条,极端情况下堆 ~250MB。`totalDropped` 仅在满时打印,无主动降载策略。

**方案**:
1. **新增堆外转储开关**:当 queue.size > `highWaterMark` (默认 20000) 时,序列化到磁盘 `.log/fallback-queue/<topic>-<key>.bin`,drainer 优先消费内存,内存空再读磁盘。
2. **降级为磁盘 mmap**:复用 JDK 21+ `MemorySegment` 写文件,读取时用 `FileChannel.map` 顺序回放,无需反序列化开销。
3. **熔断保护**:队列 > `criticalThreshold` (默认 45000) 时,producer 端直接丢弃并 `log.error` + Micrometer counter,不再 offer。

**改动文件**:
- `collector/KafkaFallbackQueue.java` — 增加 `trySpillToDisk` / `drainFromDisk` 方法
- `application.yml` — 新增 `spring-watch.kafka.fallback-queue.high-watermark`、`disk-spill-enabled`

**验证**:
- 关闭 Kafka,持续 1 分钟灌 60K 条,观察 heap 维持在 < 100MB
- Micrometer `spring.watch.collector.kafka.fallback.spill_count` 计数器 > 0

### P0-2 SelfMonitor 内存优化

**问题**:`SelfMonitorCollector.sample()` 每 10s 调用 `meterRegistry.getMeters()`,把所有 Counter/Timer/Gauge 复制到 `LinkedHashMap`。每 sample 完整复制一次,ring 保留 360 个 = 36 倍放大。

**方案**:
1. **指标白名单 + 采样复用**:
   - 维护 `Set<String> meterNameWhitelist`(只关心 `spring.watch.*` 前缀),白名单内才进入 MeterSnap
   - **每 sample 只存差量**:同一 meter 的当前值,与上一次对比,若未变则存引用;若变才存新值
2. **降采样存储**:
   - 10s 采样 → 30s 中位值 → 5min 降采样(参考 InfluxDB 降采样逻辑)
   - 1h ring 缩到 ~120 条(每 30s 一条)
3. **按需加载 Gauge/Timer 详情**:
   - `latest()` / `window()` 返回时,如果 `meters` 字段为 `null` 引用,实时从 `meterRegistry.find()` 查
   - `Sample` record 拆为 `Header(元数据) + MetersRef(实时引用)`

**改动文件**:
- `monitor/SelfMonitorCollector.java` — `captureMeters` 改用白名单 + 增量存储
- `monitor/SelfMonitorController.java`(若需要) — 暴露 `window` 接口走 `findMeter`

**验证**:
- 长跑 1h,`ring` 元素平均大小下降 50%
- heap 占用降 30~40MB

### P0-3 Log Dedup 1 次 RTT 化

**问题**:`LogDedupService.shouldKeep` 命中重复指纹时,执行 4 次 Redis 操作(setnx/incr/expire/sadd),网络 RTT + 临时对象翻 4 倍。

**方案**:**1 个 Lua 脚本原子完成**。
```lua
-- KEYS[1]=setnx key, KEYS[2]=count key, KEYS[3]=dirty set
-- ARGV[1]=window seconds, ARGV[2]=count ttl seconds, ARGV[3]=appid:fp
local first = redis.call('SET', KEYS[1], '1', 'EX', tonumber(ARGV[1]), 'NX')
if first then
  return 1  -- 保留
end
redis.call('INCR', KEYS[2])
redis.call('EXPIRE', KEYS[2], tonumber(ARGV[2]))
redis.call('SADD', KEYS[3], ARGV[3])
return 0  -- 丢弃
```

**改动文件**:
- `ingest/LogDedupService.java` — 用 `DefaultRedisScript<List>` 替换 4 次调用

**验证**:
- 10K log/s 灌入,Redis CPU 下降 50%
- Micrometer `spring.watch.ingest.log.dedup.rtt` 计时下降 75%

### P0-4 HTTP 响应流式解析

**问题**:`AgentHttpClient.get` 用 `BodyHandlers.ofString()` 一次性加载整段 Prometheus 文本。`AgentMetricsCollector` 再 `body.lines()` 流式解析,理论上 0 拷贝,但响应体已驻留堆。

**方案**:
1. **改用 `BodyHandlers.ofInputStream()` + 逐行解析**:
   ```java
   HttpResponse<InputStream> resp = httpClient.send(req, BodyHandlers.ofInputStream());
   try (BufferedReader br = new BufferedReader(new InputStreamReader(resp.body(), UTF_8))) {
       br.lines().filter(...).forEach(...);
   }
   ```
2. **指标 + 日志两处都改**:
   - `AgentMetricsCollector.collect` 接收 `InputStream`,不再接收 `String`
   - `AgentLogCollector.collect` 接收 `InputStream`,用流式 JSON 解析(jackson `JsonParser` token 模式,不整体 `readValue`)
3. **大小硬上限**:加 `Content-Length` 检查,> 5MB 直接拒绝(`result.failed("body_too_large")`)

**改动文件**:
- `collector/AgentHttpClient.java` — 新增 `getStream(url)` 方法
- `collector/AgentMetricsCollector.java` — `collect` 接收 `InputStream`
- `collector/AgentLogCollector.java` — `collect` 接收 `InputStream`,用 jackson streaming

**验证**:
- 1000 app × 1MB scrape,heap 占用降 50~100MB
- OOM 风险消除

### P0-5 Prometheus 解析零分配

**问题**:`parsePrometheusLine` 每次 `new HashMap<>()` 存 tags,200 行 = 200 HashMap,GC 压力大。

**方案**:**复用 `ThreadLocal<Map<String, String>>`,每次解析前 clear**。
```java
private static final ThreadLocal<Map<String, String>> TAG_MAP = ThreadLocal.withInitial(HashMap::new);

ParsedMetric parsePrometheusLine(String line) {
    ...
    Map<String, String> tags = TAG_MAP.get();
    tags.clear();
    // 解析 tags 进 tags
    return new ParsedMetric(metricName, tags, value);
}
```

**注意**:
- `ParsedMetric` 持有 tags 引用 → 调用方(`AgentMetricsCollector.toMetricEvent`)用完即弃,不可传给其他线程
- `MetricEvent.tags` 在序列化时由 Jackson 复制,不会保留引用
- 风险点:`toMetricEvent` 当前直接 `.tags(parsed.tags)`,确认下游不会跨线程缓存

**改动文件**:
- `collector/AgentMetricsCollector.java` — ThreadLocal 改造

**验证**:
- jcmd `jcmd <pid> GC.class_histogram | grep HashMap` 计数下降
- 1000 app scrape 时 Young GC 频率下降 30%+

### P1-1 MonitorApp 内存缓存

**问题**:`AppPullTask.run` + `doHeavyWork` 每次拉取 2 次 `findByAppid`,高频 DB 命中。

**方案**:**Caffeine 本地缓存(2 分钟过期,与 scrape 周期对齐)**。
```java
@Component
public class MonitorAppCache {
    private final Cache<Long, MonitorApp> cache = Caffeine.newBuilder()
        .expireAfterWrite(Duration.ofMinutes(2))
        .maximumSize(10_000)
        .build();
    public Optional<MonitorApp> get(Long appid) { ... }
    public void invalidate(Long appid) { ... }
}
```

**失效时机**:`MonitorAppController` 增删改时调用 `invalidate(appid)`。

**改动文件**:
- 新增 `service/MonitorAppCache.java`
- `collector/AppPullTask.java` — 替换 `repository.findByAppid` 为 `cache.get`
- `web/MonitorAppController.java` — 增删改时 `cache.invalidate`

**验证**:
- DB QPS 降 80%+
- heap 增加 < 5MB

### P1-2 LogSanitizer 合并正则

**问题**:`LogSanitizer.mask` 对 message/throwable 链式应用 5 个 Pattern,每次 `replaceAll` 分配新 String。

**方案**:**1 个复合 Pattern,用 `Matcher` 一次扫描 + 分支替换**。
```java
private static final Pattern SENSITIVE = Pattern.compile(
    "(?<phone>1[3-9]\\d{9})|(?<idcard>\\d{17}[\\dXx])|(?<bank>\\d{16,19})|" +
    "(?<email>[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,})|" +
    "(password|passwd|pwd|secret|token)([\"']?\\s*[:=]\\s*[\"']?)([^\\s,;'\"}]+)"
);
```

**注意**:复合正则需要 lookahead 排除部分误匹配(原 Pattern 用 `(?<![\d])` 边界,需在 Matcher 中 `region` 处理)。

**改动文件**:
- `ingest/LogSanitizer.java` — 合并为 1 个复合 Pattern

**验证**:
- 单元测试覆盖相同脱敏用例
- 性能:10K log/s 灌入,GC 分配速率降 20%+

### P1-3 LogFingerprinter MessageDigest 复用

**问题**:`MessageDigest` 非线程安全,且每次 `getInstance` + `digest` 都分配 byte[20]。

**方案**:**ThreadLocal<MessageDigest>**,每次 reset 后复用。
```java
private static final ThreadLocal<MessageDigest> SHA1 = ThreadLocal.withInitial(() -> {
    try { return MessageDigest.getInstance("SHA-1"); }
    catch (NoSuchAlgorithmException e) { throw new IllegalStateException(e); }
});

private String sha1Hex(String input) {
    MessageDigest md = SHA1.get();
    md.reset();
    byte[] hash = md.digest(input.getBytes(UTF_8));
    return HexFormat.of().formatHex(hash);
}
```

**改动文件**:
- `ingest/LogFingerprinter.java` — ThreadLocal MessageDigest

**验证**:
- jcmd `MessageDigest` 实例计数恒为 1
- 10K log/s 时分配速率下降

### P1-4 AlertEngine 合成指标零分配

**问题**:`AlertEngine.toSyntheticMetric` 每次 new HashMap,日志类规则命中时高分配。

**方案**:**复用 `ThreadLocal<Map<String, String>>`,与 P0-5 同模式**。
- 调用方:仅在 `evaluateLogRule` 内单次使用,序列化后即弃,安全
- 加 `try { ... } finally { tags.clear(); }` 确保清空

**改动文件**:
- `alerter/AlertEngine.java`

**验证**:
- 长跑告警命中 1K/s 时 Young GC 频率下降

### P1-5 AlertStateStore.scanPending 减少 entries() 调用

**问题**:每次 scan 对每个 PENDING key 调用 `opsForHash().entries()`,返回完整 Map。10K PENDING = 10K 次 `HGETALL`。

**方案**:
1. **HSCAN 替代 HGETALL**:
   - 用 `redis.opsForHash().scan(key, ScanOptions)` 流式读,只读需要字段
2. **字段精简**:
   - 扫描器只关心 `state`、`firstBreachAt`、`triggerCount`,不读 `lastValue` / `lastMetric` / `lastTags`
3. **Pipelining 批量**:
   - `executePipelined` 一次 N 个 HGET,减少 RTT

**改动文件**:
- `alerter/AlertStateStore.java` — `scanPending` 改用 HSCAN + pipeline

**验证**:
- 1K PENDING 时单次扫描耗时降 50%+
- Redis CPU 下降

### P1-6 SelfMonitor.window() 异步加载

**问题**:`window()` 用 synchronized 阻塞,大窗口查询时阻塞采样线程。

**方案**:
1. **CopyOnWrite 模式**:
   - sample 写入 `Deque<SampleRef>`,ref 持轻量 `Sample` 引用
   - 读时返回 `List<SampleRef>`,按需 `ref.materialize()` 触发当前 `meterRegistry` 实时查询
2. **或简单方案**:`synchronized` 范围缩到只 `addLast/peekLast`,`window` 加 read-write lock

**改动文件**:
- `monitor/SelfMonitorCollector.java`

**验证**:
- 监控页 60s 刷新,不再影响采样

## 5. JVM 调优(配套)

> 不修改 SpringBoot 版本和 JVM 版本,仅加启动参数。

`application.yml` / 环境变量 `JAVA_OPTS` 建议:
```bash
# 堆:典型 1.5G,峰值 2G
-Xms1g -Xmx2g
# G1 调优(Java 25 默认 G1)
-XX:+UseG1GC
-XX:MaxGCPauseMillis=100
-XX:G1HeapRegionSize=16m
# 虚拟线程(Spring Boot 4 默认启用,显式确认)
-Djdk.virtualThreadScheduler.parallelism=$(nproc)
-Djdk.virtualThreadScheduler.maxPoolSize=256
# 字符串去重(降低堆占用,Java 25 增强)
-XX:+UseStringDeduplication
# 容器感知(若 k8s 部署)
-XX:+UseContainerSupport
-XX:MaxRAMPercentage=75.0
```

## 7. 风险与边界

### 7.1 兼容性约束(来自白皮书)

- **不修改 Spring Boot 版本和 JVM 版本**(用户约束)→ 仅改代码与启动参数
- **拉取模型不变**(白皮书约束 1)→ 不引入 push / WebSocket
- **Java Agent 接入不变**(白皮书约束 4)→ 不动 mock-test,不要求客户改
- **0 复制 / 0 依赖**(白皮书 §6)→ 所有优化均在平台侧

### 7.2 风险项

| 风险 | 缓解 |
|---|---|
| ThreadLocal 内存泄漏(线程池场景) | 用 `WeakReference` 或 `cleaner` 机制;`@Scheduled` 任务在退出时显式 `remove()` |
| HTTP 流式改造对 body 大小不敏感(本来 ofString 显式撑爆) | 加 `Content-Length` 硬上限 + 超时熔断 |
| SelfMonitor 改造影响监控页 | 前端走 `latest()` 实时拉,`window()` 异步,接口文档同步更新 |
| 磁盘降载依赖文件系统 | 容器化部署需 mount `.log` volume;失败时降级为纯内存 + 更激进丢弃 |
| Caffeine 缓存失效不及时 | 加 actuator 端点手动清,Prometheus 告警 `cache hit rate < 80%` |

### 7.3 不做的事(明确边界)

- ❌ 不引入新依赖(无 Caffeine / Caffeine 已存在除外,需评估)
- ❌ 不把 Prometheus 解析换库(原 split-based 自写解析足够)
- ❌ 不做分布式内存聚合(单实例优化)
- ❌ 不改数据模型(白皮书 §8 不可变)
- ❌ 不引入 Spark/Flink 等大数据组件
