> 输入:`docs/metrics.txt` (mock-test 真实暴露的 Prometheus 文本)
> 目标:把 JDBC / HTTP / JVM / OS 四大类指标接进渲染层(应用详情页 4 个 Tab)
> 时间:2026-06-19

## 1. 指标清单(metrics.txt 实测)

按 Prometheus `metric_name` 前缀聚类,共 **30** 个原生指标 + 1 个 `target_info` 元数据:

| 大类 | 指标前缀 | 数量 | 实例(Otel instrumentation) |
|---|---|---|---|
| **JDBC** | `db_client_connections_*` | 7 | HikariCP-3.0 (OTel 自动) |
| **HTTP** | `http_server_request_duration_seconds` | 1 | tomcat-10.0 (OTel 自动) |
| **JVM** | `jvm_*` | 12 | runtime-telemetry-java8 (OTel 自动) |
| **OS / Runtime** | `system_*` / `runtime_*` | 9 | oshi (OTel 自动) |

> ⚠️ **与 `metric_dict` 种子表命名差异**:
> - 字典表登记的是 Micrometer 1.x 时代的 `hikaricp_*` / `jdbc_*` / `process_*` 旧名
> - mock-test 实际跑的是 OTel Javaagent 2.28.1,出来的是 `db_client_*` / `runtime_*` / `system_*` 新名
> - 本期不修字典,渲染层直接按 **metrics.txt 的真实名** 渲染,字典 v2 再做映射

### 1.1 JDBC (7)

| 指标 | 类型 | 标签 | 含义 |
|---|---|---|---|
| `db_client_connections_create_time_milliseconds` | histogram | pool_name | 创建新连接耗时分布 |
| `db_client_connections_idle_min` | gauge | pool_name | 最小空闲连接 |
| `db_client_connections_max` | gauge | pool_name | 最大连接数 |
| `db_client_connections_pending_requests` | gauge | pool_name | 等待连接的请求数 |
| `db_client_connections_usage` | gauge | pool_name, **state**(idle/used) | 各状态连接数 |
| `db_client_connections_use_time_milliseconds` | histogram | pool_name | 连接使用时长(借→还) |
| `db_client_connections_wait_time_milliseconds` | histogram | pool_name | 获取连接等待时长 |

### 1.2 HTTP (1,懒加载)

| 指标 | 类型 | 标签 | 含义 |
|---|---|---|---|
| `http_server_request_duration_seconds` | histogram | method, status_code, route, protocol_version, scheme | HTTP 请求耗时 |

> **懒加载说明**:OTel HTTP server instrumentation 只在 **真正发生过请求** 的 `(method, route, status)` 组合上写指标。
> 当前 mock-test 只触发过 `GET /api/ping` 1 次,所以文件里只有这一条组合。**渲染层必须容忍"指标为空"** —— 显示"暂无请求,先调用任意 API 触发采集"提示。

### 1.3 JVM (12)

| 指标 | 类型 | 标签 | 含义 |
|---|---|---|---|
| `jvm_class_count` | gauge | — | 当前加载类数 |
| `jvm_class_loaded_total` | counter | — | 累计已加载类 |
| `jvm_class_unloaded_total` | counter | — | 累计已卸载类 |
| `jvm_cpu_count` | gauge | — | CPU 核心数 |
| `jvm_cpu_recent_utilization` | gauge | — | CPU 使用率(0~1) |
| `jvm_cpu_time_seconds_total` | counter | — | 累计 CPU 时间(秒) |
| `jvm_gc_duration_seconds` | histogram | jvm_gc_action, jvm_gc_name | GC 耗时 |
| `jvm_memory_committed_bytes` | gauge | jvm_memory_pool_name, jvm_memory_type | 已提交内存 |
| `jvm_memory_limit_bytes` | gauge | jvm_memory_pool_name, jvm_memory_type | 内存上限 |
| `jvm_memory_used_after_last_gc_bytes` | gauge | jvm_memory_pool_name, jvm_memory_type | GC 后内存 |
| `jvm_memory_used_bytes` | gauge | jvm_memory_pool_name, jvm_memory_type | 已用内存 |
| `jvm_thread_count` | gauge | jvm_thread_daemon, jvm_thread_state | 各状态线程数 |

### 1.4 OS / Runtime (9)

| 指标 | 类型 | 标签 | 含义 |
|---|---|---|---|
| `runtime_java_cpu_time_milliseconds` | gauge | type(system/user) | Java 进程 CPU 时间 |
| `runtime_java_memory_bytes` | gauge | type(rss/vms) | Java 进程内存(物理/虚拟) |
| `system_disk_io_bytes_total` | counter | device, direction(read/write) | 磁盘 IO 字节数 |
| `system_disk_operations_total` | counter | device, direction | 磁盘操作数 |
| `system_memory_usage_bytes` | gauge | state(free/used) | 系统内存使用(字节) |
| `system_memory_utilization` | gauge | state(free/used) | 系统内存使用率(0~1) |
| `system_network_errors_total` | counter | device, direction | 网卡错误数 |
| `system_network_io_bytes_total` | counter | device, direction | 网卡 IO 字节数 |
| `system_network_packets_total` | counter | device, direction | 网卡包数 |

### 1.5 InfluxDB 落库结构(`springboot_metrics` measurement)

OTel 推过来的指标,在 `BatchMetricConsumer` 里被转成 InfluxDB Point:

```
measurement: springboot_metrics
tags:    appid, metric, method, <Prometheus 原 labels 平铺>
fields:  value(double), count(long, 可选)
time:    NS 精度
```

> Prometheus 里的 `db_client_connections_usage{state="idle"}` →
> InfluxDB 里 `springboot_metrics{appid=xxx, metric="db_client_connections_usage", state="idle"} value=10`
>
> histogram(`*_bucket/_sum/_count`)被展成 3 个独立 metric 写入,后缀保留:
> `db_client_connections_create_time_milliseconds_bucket` / `_sum` / `_count`

## 2. 渲染总策略

### 2.1 落地位置

```
src/main/resources/static/
├── index.html              # 壳页面
├── css/style.css           # 单一全局样式
├── js/common.js            # SW 全局对象(API + ECharts 包装)

    └── app-detail.html     # 4 大 Tab 的唯一渲染页
```

> 最小可用版本:**只做 `app-detail.html` + 必需公共件**。其余页面(apps.html / alert-* 等)后续按 `视图层规划_v1.md` 渐进补。

### 2.2 路由与上下文

- 壳 `index.html` 加载,默认 iframe 指向 `pages/dashboard.html`(v1 暂用 `app-detail.html` 占位,跳转到第一个 active 应用)
- 顶栏有**应用选择器** `<select id="app-switcher">`,绑 `localStorage.currentAppid`
- 业务页 `?appid=xxx` 优先,否则回退到 `localStorage.currentAppid`,都没有 → 提示"请先选择应用"
- 跨域:零。`http://localhost:8080/` 同源,fetch 直接打 `/api/*`

- 默认 30s 轮询当前 Tab
- 提供"暂停"按钮,避免调试时数据跳

## 3. 各 Tab 详细设计

### 3.1 JDBC Tab

| 区块 | 指标 | 渲染方式 | 备注 |
|---|---|---|---|
| 数值卡片×4 | `db_client_connections_max`、`db_client_connections_idle_min`、`db_client_connections_usage{state=idle}`、`db_client_connections_usage{state=used}` | `numberCard` | "使用率"卡片算 `used / max * 100%` |
| 等待连接 | `db_client_connections_pending_requests` | `numberCard`(大字号 + 阈值色:>0 红) | 排队请求要醒目 |
| 池内连接数时序 | `db_client_connections_usage{state=idle\|used}` 堆叠面积 | `line({stack:true, area:true})` | 看 idle/used 占比变化 |
| 创建连接耗时 P50/P95 | `db_client_connections_create_time_milliseconds_bucket` 算 quantile(取 bucket 近似) | `line` | 懒触发,首次可能空 |
| 连接使用时长分布 | `db_client_connections_use_time_milliseconds_bucket` | `bar`(le 区间作为 x 轴) | 关注长尾 |
| 等待连接耗时分布 | `db_client_connections_wait_time_milliseconds_bucket` | `bar` | 长尾 = 池子吃紧 |

**后端需要**:
- 卡片:`/api/metrics/latest?appid=&metric=&pool_name=&state=`
- 时序:`/api/metrics/series?appid=&metric=&from=&to=&agg=mean&every=30s`
- 直方图还原 P50/P95:用 `_bucket` + `_count` 重算 → `/api/metrics/histogram-quantile?appid=&metric=&quantile=0.5,0.95,0.99`

### 3.2 HTTP Tab

| 区块 | 指标 | 渲染方式 | 备注 |
|---|---|---|---|
| 空状态 | 无任何 `http_server_request_duration_seconds_*` | 提示卡片:"暂无请求,先去 mock-test 点几下接口触发采集" | 懒加载,必须有 |
| 请求 QPS 时序 | `http_server_request_duration_seconds_count` rate 一下 | `line` | unit=req/s |
| 响应时间 P50/P95/P99 | 同 §3.1 的 histogram-quantile | `line`(3 series) | 关注 P99 |
| 状态码分布 | 按 `http_response_status_code` group 后 last value | `pie` | 200/4xx/5xx |
| 慢请求 TopN | 按 `http_route` 排序,取 sum 最大的前 5 | `bar(horizontal)` | 表格替代也行 |
| 路由×状态 热力 | `route × status` 计数 | `heatmap` | v2 再说 |

**懒加载容错**:
- 后端查空时返回 `{empty: true, hint: "..."}`
- 前端按 `empty` 显空态卡,不报错不破图

### 3.3 JVM Tab

| 区块 | 指标 | 渲染方式 |
|---|---|---|
| 数值卡片×4 | `jvm_cpu_recent_utilization`、`jvm_class_count`、`jvm_class_loaded_total`(累计)、活跃线程数(从 `jvm_thread_count{jvm_thread_state="runnable",jvm_thread_daemon="false"}` 取) | `numberCard` |
| 堆内存使用 | `jvm_memory_used_bytes{jvm_memory_type="heap"}` 按 `jvm_memory_pool_name` 堆叠 | `line({stack:true, area:true})` |
| 非堆内存使用 | 同上,`jvm_memory_type="non_heap"` | `line({stack:true, area:true})` |
| 内存使用率 | `used / limit` 按 pool | `line` |
| GC 暂停耗时 | `jvm_gc_duration_seconds_sum` / `_count` → 平均暂停 | `bar`(按 `jvm_gc_name`) |
| GC 次数时序 | `jvm_gc_duration_seconds_count` | `line` |
| 线程状态分布 | `jvm_thread_count` 按 `jvm_thread_state` 分组,取最新值 | `pie` |
| 类加载累计 | `jvm_class_loaded_total` / `jvm_class_unloaded_total` | `line`(双 y 轴) |

### 3.4 OS Tab

| 区块 | 指标 | 渲染方式 |
|---|---|---|
| 数值卡片×4 | `jvm_cpu_count`、`system_memory_utilization{state="used"}`、`runtime_java_memory_bytes{type="rss"}`(人类可读 MB/GB)、`runtime_java_cpu_time_milliseconds{type="user"}` | `numberCard` |
| 系统内存使用 | `system_memory_usage_bytes{state=used}` 时序 + 总内存(`used+free`)作参考 | `line({stack:true})` |
| 系统内存使用率 | `system_memory_utilization{state=used}` | `line({area:true})` |
| 进程 CPU 时间累计 | `runtime_java_cpu_time_milliseconds` 按 `type`(user/system) | `line`(双 y 轴) |
| 进程内存 | `runtime_java_memory_bytes{type=rss\|vms}` | `line`(双 y 轴) |
| 磁盘 IO 字节 | `system_disk_io_bytes_total` 按 `device` × `direction` 堆叠 | `line({stack:true})` |
| 磁盘 IO 操作 | `system_disk_operations_total` 同上 | `line` |
| 网卡 IO | `system_network_io_bytes_total` 按 `device` × `direction` | `line`(双 y 轴) |
| 网卡包数 / 错误数 | `system_network_packets_total` / `system_network_errors_total` | `bar(horizontal)`(按 device 聚合) |

## 4. 后端 API 设计

> 现状:`MetricController` / `MetricQueryService` 是空壳,需要全部补齐。
> 存储:`springboot_metrics` measurement(详见 §1.5)

### 4.1 端点清单

| 方法 | 路径 | 用途 | 入参 | 出参 |
|---|---|---|---|---|
| GET | `/api/metrics/available` | 列出某 app 实际出现过的指标 | `appid` | `[{metric, count, lastSeen, tags}]` |
| GET | `/api/metrics/latest` | 某 metric 的最新点(可按 tag 过滤) | `appid, metric, pool_name?, state?, jvm_memory_pool_name?, ...` | `{value, time, tags}` |
| GET | `/api/metrics/series` | 时序数据 + 聚合 | `appid, metric, from, to, agg(mean/max/last), every(30s), <tag 过滤>` | `[[t, v], ...]` |
| GET | `/api/metrics/grouped` | 按某 tag 分组的最新快照 | `appid, metric, groupBy` | `[{group: "idle", value: 10}, ...]` |
| GET | `/api/metrics/histogram-quantile` | 直方图还原 P50/P95/P99 | `appid, metric, quantile=0.5,0.95,0.99, from, to, every(30s)` | `[[t, q50, q95, q99]]` |
| GET | `/api/apps/active` | 应用切换器数据源 | — | `[MonitorApp]` |
| GET | `/api/apps/{id}` | 单个应用详情 | id | `MonitorApp` |
| GET | `/api/apps` | 列表(含暂停) | — | `[MonitorApp]` |

### 4.2 Flux 查询模板

**latest**:
```flux
from(bucket: "metrics")
  |> range(start: -1m)
  |> filter(fn: (r) => r._measurement == "springboot_metrics"
                    and r.appid == "<appid>"
                    and r.metric == "<metric>"
                    and r.<tagKey> == "<tagVal>")
  |> last()
  |> keep(columns: ["_value", "_time", "<其它 tag>"])
```

**series**:
```flux
from(bucket: "metrics")
  |> range(start: <from>, stop: <to>)
  |> filter(fn: (r) => r._measurement == "springboot_metrics"
                    and r.appid == "<appid>"
                    and r.metric == "<metric>")
  |> aggregateWindow(every: <every>, fn: <agg>, createEmpty: false)
  |> yield(name: "series")
```

**histogram-quantile**(简化版,线性插值):
- 后端拉 `_<metric>_bucket{le="..."}` + `_<metric>_count`
- 按 `le` 排序,取 `q * count` 对应的桶,反推 `le` 边界值
- 实现参考 Prometheus `histogram_quantile()` 线性插值

### 4.3 数据格式约定

- 时间一律 `Instant` → ISO-8601 字符串(前端 ECharts time 轴直接吃)
- 数值 `Double`,NaN/Infinity → `null`(前端要容忍)
- 标签存 InfluxDB tag,值是 string,前端不过滤空白
- 列表接口外层包 `ApiResponse.ok(data)`

### 4.4 Micrometer 埋点

- `spring.watch.metric.query.latest` / `.series` / `.grouped` / `.histogram` Timer
- `spring.watch.metric.query.fail` Counter
- `spring.watch.metric.query.empty_result` Counter(区分数据真零 vs 没采集到)

## 5. 前端架构

### 5.1 工具链

- **零构建**:HTML5 + 原生 JS + ECharts 5.5 CDN
- **路由**:iframe + 左侧导航(顶栏放应用切换器)
- **HTTP**:`fetch` 走 `SW.api.xxx`
- **样式**:单一 `style.css`,CSS 变量定义色板
- **图标**:emoji(不引外部图标库)

### 5.2 `SW` 全局对象

```js
window.SW = {
  api: {
    get(url, params) -> Promise<data>      // 自动 ApiResponse 解包
    post(url, body) -> Promise<data>
    put(url, body) -> Promise<data>
    del(url) -> Promise<data>
  },
  ctx: {
    getAppid() -> string|null              // 优先 ?appid,回退 localStorage
    setAppid(appid)
    listApps() -> Promise<MonitorApp[]>
  },
  ui: {
    toast(msg, type)                       // info/success/warn/error
    formatBytes(n) -> "1.23 MB"            // OS Tab 用
    formatPercent(n, fixed=1) -> "12.3%"
    formatNumber(n) -> "1,234,567"
  },
  chart: {
    line(el, data, opts)                   // data=[{name, points:[[t,v]]}]
    bar(el, data, opts)                    // data={categories, series:[{name,data}]}
    pie(el, data, opts)                    // data=[{name,value}]
    numberCard(el, value, opts)            // 数值卡
    destroy(el)
  }
}
```

### 5.3 ECharts 公共配置

- 时间轴 `xAxis.type = 'time'`,统一 UTC
- 颜色:`['--c-primary', '--c-warn', '--c-success', '--c-danger', '--c-info']` 顺序取
- tooltip.trigger = 'axis',带 crosshair
- 内置 dataZoom(滑动条 + 滚轮缩放)
- `window.resize` 自动 `chart.resize()`

### 5.4 空状态容错(懒加载关键)

- 卡片无数据:`<div class="empty-card">暂无数据</div>`(灰色)
- Tab 完全无数据(典型如 HTTP):Tab 内顶部 `alert-info`,提示"先去 mock-test 触发接口"
- 接口 404 / 500:`toast.error(...)` + 控制台 warn

## 7. 反模式(防止返工)

- ❌ 不用 Vue/React/npm/webpack
- ❌ 不在 `common.js` 之外写 `fetch`(统一拦截器无法兜底)
- ❌ 不把 ECharts option 散落到各页面(从 `SW.chart` 出)
- ❌ 不写"现在没数据"的 JS 报错分支(接口正常返回空数组就是空数组)
- ❌ 不在 SQL 里查 `metric_dict` 拼映射(字典 v2 再做,渲染层先吃真实名)
- ❌ 不做权限 / 鉴权(v1 平台内网用)
- ❌ 不引 jQuery / lodash
