## 二、采集层(Collector + Consumer + Ingest)

### 2.2 缺失清单(按"生产可用"标准)

#### P0 - 阻断性

| # | 缺失 | 现状 | 影响 |
|---|---|---|---|
| 2 | **无 HTTP 连接池** | `AgentMetricsCollector` / `AgentLogCollector` 每次 `new HttpURLConnection` | 高并发拉取时 TPC 连接耗尽、握手开销巨大 |
| 4 | **agent 失活无独立探测线程** | 仅靠下次 `AppPullTask` 拉取时 `isReachable` 发现,间隔最长 `scrapeInterval`(默认 15s) | agent 挂掉可能要 15s+ 才感知,且期间不更新 `lastHeartbeat`,没有"心跳超时"主动告警 |
| 5 | **PullRetryQueue 无容量上限** | `PriorityBlockingQueue` 无限,所有 retry 任务都堆内存 | agent 全挂时 OOM 风险 |
| 6 | **没有指标采集维度补全** | `MetricEvent.tags` 仅 Prometheus 自带 + source/statusCode,无 env/labels | 跨应用聚合/分组能力弱 |

#### P1 - 短板

| # | 缺失 | 现状 | 影响 |
|---|---|---|---|
| 9 | **AgentLogCollector 无响应大小限制** | `BufferedReader` 一行读到 EOF | agent 返回 1GB 会 OOM 平台 |
| 10 | **无采集层指标(Micrometer)** | 没有 `collect_total` `collect_duration` `collect_fail` 等埋点 | 无法监控"每秒拉多少次/失败率" |
| 12 | **采集层无 restart-safe** | 调度全在内存,重启后从 DB 重建(`@PostConstruct`) | 重启瞬间有数据空窗,且 lastLogPullTime 不会回退,可能漏窗口 |
| 13 | **`AppSchedule.firstDelayMs` jitter 范围** | `jitter-percent: 10`,计算固定 | 大量应用同一时刻启动还是会"雪崩" |
| 14 | **agent 版本/能力探测** | `HeartbeatEvent.agentVersion="java-agent"` 写死 | 多 agent 版本兼容、灰度升级无法做 |
| 15 | **endpoint 没有 TLS/HTTPS 校验** | `HttpURLConnection` 默认 trust all | 接入公网应用有 MITM 风险 |

### 3.2 缺失清单

#### P0 - 阻断性

| # | 缺失 | 现状 | 影响 |
|---|---|---|---|
| 1 | **InfluxDB 无 retention policy** | 启动只创建 bucket,没设保留时长,数据**永久保留** | 生产跑 1 个月就撑爆磁盘,**最致命的隐患** |
| 2 | **无 InfluxDB downsample** | 没有 task/downsample rule 把 1m 数据降为 1h/1d | 高频指标永远全量保留,成本失控 |
| 3 | **PostgreSQL 无业务索引** | `alert_history` 只有 PK + FK,`alert_rule` 同样 | "按 appid 查告警历史"会全表扫,告警多了直接卡死 |
| 4 | **InfluxDB 高基数 tag 无防护** | `fingerprint` 作为 tag,理论上有 100w+ 唯一值 | InfluxDB 索引爆,写入退化甚至 OOM |
| 5 | **AlertHistory 没有 appid 字段冗余** | 只有 `app` LAZY 关联,`findByAppid` 要 join | 告警历史查询慢 |
| 6 | **无数据归档/冷热分层** | 监控数据全在 InfluxDB 一层 | 长期数据查询慢、磁盘压力大 |

#### P1 - 短板

| # | 缺失 | 现状 | 影响 |
|---|---|---|---|
| 7 | **InfluxDB queryApi 无池化** | `MetricQueryService` / `LogQueryService` / `LogAggregator` 每次 `influxDBClient.getQueryApi()` | 大量并发查询会建一堆 HTTP 连接 |
| 8 | **AlertHistory 没有 updatedAt** | 只有 createdAt + resolvedAt | 无法查"最近 N 分钟触发"类查询 |
| 9 | **alert_rule 无 status 索引** | status 用作 WHERE 条件但无索引 | 规则多了 `findByStatus` 慢 |
| 10 | **monitor_app 无 createdAt 索引** | 业务上"最近注册"查询慢 |
| 11 | **DLQ 落库表不存在** | DLQ 仅日志,无 `dlq_message` 表 | DLQ 消息无法重投/查询 |
| 12 | **无数据备份策略** | InfluxDB / PostgreSQL 无文档化的备份方案 | 故障恢复没保障 |
| 13 | **InfluxDB 写失败 fallback 不全** | 写失败抛 RuntimeException 整批回滚 | 偶尔一条格式错就丢一整批 |
| 14 | **无连接池配置(max connections)** | InfluxDB Java Client 默认连接数小 | 高并发查询瓶颈 |
| 15 | **无多租户隔离** | 单 org / 单 PG schema | 上云后无法服务多个客户 |

#### P2 - 增强

| # | 缺失 |
|---|---|
| 16 | Schema registry(metric_name / label 黑白名单) |
| 17 | 冷数据归档到 OSS/S3 |
| 18 | 数据压缩(Parquet 列存归档) |
| 19 | 数据库迁移的 rollback(V1-V8 只有 forward) |

## 六、跨四模块的"非功能性"缺口

| # | 项 | 现状 | 影响 |
|---|---|---|---|
| 1 | **无 Metrics/Micrometer 埋点** | spring-watch 自己没有任何自我监控指标 | 平台"自监控"全靠看日志,无法告警自身 |
| 2 | **无分布式追踪** | 业务侧 `LogEvent.traceId` 仅透传,平台内部不生成 | 排查"为啥告警慢"无从下手 |
| 3 | **无审计日志** | 谁注册 app / 谁改规则 / 谁删 app 无记录 | 合规性问题 |
| 4 | **无配置中心** | `application.yml` 写死,SMTP 密码明文 | 切换环境/热更配置要走重启 |
| 5 | **无灰度/金丝雀** | 新规则一旦 enable 立即全量生效 | 误配规则会刷屏 |
| 6 | **无多实例支持** | `AlertStateStore` 用 Redis,理论上可多实例,但调度层(CollectScheduleRegistry)是多实例时会重复拉 | 横向扩展会重复采集 |
| 7 | **无 graceful shutdown 完整性** | `KafkaFallbackQueue` / `CollectScheduleRegistry` 有 `@PreDestroy`,但 `BatchMetricConsumer` / `BatchLogConsumer` 没有显式 drain | 进程重启会丢正在写库的那批 |
| 8 | **无 OpenAPI/Schema 文档** | 没有 springdoc,前端只能靠口口相传 | 联调成本高 |
| 9 | **无单元测试覆盖率统计** | docs 提到 alerter 有 48 个测试,但没看到 `jacoco` 配置 | 不知道核心逻辑测试覆盖度 |
| 10 | **无负载/压力测试报告** | docs/限流测试计划.md 提到但没看到执行结果 | 不知道当前架构能撑多少 app |
