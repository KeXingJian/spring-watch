## 一、什么是高可用(High Availability, HA)

### 1.1 一句话定义

> **高可用 = 系统在可观测的时间窗口内,持续对外提供服务的能力。**
>
> 它衡量的不是"会不会出故障",而是"出故障后多久能恢复、用户感知到什么"。

### 1.2 三个核心要素

| 要素 | 含义 | 衡量指标 |
|------|------|----------|
| **可靠性(Reliability)** | 在给定条件下、按设计功能正常运行的概率 | MTBF(平均故障间隔时间) |
| **可维护性(Maintainability)** | 故障发生后恢复到正常状态的难度 | MTTR(平均恢复时间) |
| **冗余度(Redundancy)** | 关键组件是否有备份、能否承受 n-1 故障 | RPO / RTO |

业界把"高可用"量化为 **SLA(Service Level Agreement)**,常见档位:

| SLA | 年累计停机 | 适用场景 |
|-----|-----------|----------|
| 99.9% (三个 9) | 8.76 小时 | 普通业务 |
| 99.95% | 4.38 小时 | 中等业务 |
| 99.99% (四个 9) | 52.6 分钟 | 金融、监控、计费 |
| 99.999% (五个 9) | 5.26 分钟 | 电信核心网 |

> 关键洞察:**高可用是 MTTR 的函数,不是 MTBF 的函数**。把 MTTR 压到 30 秒,即便每月崩一次,也能达到 99.99%。

### 1.3 高可用的工程手段(按层次)

1. **消除单点(SPOF)**
   - 多副本(主备、主主、多活)
   - 无状态服务 + 状态外置
2. **故障检测**
   - 健康探针(Liveness / Readiness)
   - 心跳、超时、租约(Lease)
3. **自动恢复**
   - 进程崩溃 → `systemd` / `docker restart` / K8s Pod 重启
   - 节点宕机 → 流量调度到健康节点
4. **流量调度**
   - L4/L7 负载均衡、Nginx upstream、K8s Service
   - 一致性 Hash、就近路由
5. **数据不丢**
   - WAL(预写日志)、复制集(Replication)、多副本 ACK
   - 备份 + PITR(时间点恢复)
6. **优雅降级**
   - 限流、熔断、降级开关、本地缓存兜底
7. **可观测性**
   - 监控自身的监控(自监控 / Meta-monitoring)
   - 告警分级、On-call 流程

### 1.4 常见误区

- ❌ "高可用 = 分布式":分布式只是手段之一,不是充分必要条件
- ❌ "高可用 = 不停机":所有系统终将故障,关键是**用户无感**
- ❌ "高可用 = 多花钱":很多 HA 手段是软件层面的(进程级冗余、本地缓存),边际成本极低
- ❌ "高可用 = 万无一失":不存在 100% 可用,只能无限逼近 100%

## 二、非分布式能不能做高可用?

### 2.1 短答:**能**,而且**已经在做**

> 分布式是实现高可用的**一种手段**,不是**唯一手段**。

判断标准不是"是否分布式",而是回答这三个问题:

1. **故障后能否自动恢复?** (重启 / 切换)
2. **恢复过程中是否丢数据?** (持久化 / 复制)
3. **用户是否感知到中断?** (SLA)

只要三者都能给出"是"或"在 SLA 内的否",就是高可用。

### 2.2 单机(非分布式)做高可用的 7 种手段

| 手段 | 原理 | 适用 | spring-watch 现状 |
|------|------|------|------------------|
| **① 进程级多副本** | 同机起 N 个进程 + 端口分流 | 无状态服务 | ✅ 可行(Ingest 入口) |
| **② 进程级主备** | 一主一备,VIP/端口漂移 | 有状态服务 | ✅ 可行(Kafka / InfluxDB 都支持单机多实例) |
| **③ 状态外置 + 本地缓存** | 启动时拉取、运行期用 Caffeine | 状态可重建 | ✅ 已落地(`AlertStateStore` / `LogDedupService`) |
| **④ 优雅停机 + 流量摘除** | SIGTERM → 排空 → 退出 | K8s 滚动升级 | ✅ Spring Boot `GracefulShutdown` |
| **⑤ 自动重启 + 探活** | `systemd Restart=always` / `docker restart: always` | 进程崩溃 | ✅ 已在 `docker-compose.yml` 体现 |
| **⑥ 反压 + 降级** | 有界队列 + 拒绝策略 | 突发流量 | ✅ 已落地(降级队列 10k 上限) |
| **⑦ 自监控 + On-call** | 监控系统监控自己 + 告警人 | 任何规模 | ✅ 已落地(Micrometer + InfluxDB 双桶) |

### 2.3 单机高可用的**天花板**

| 失效场景 | 单机能不能扛? |
|---------|---------------|
| 进程崩溃 / OOM | ✅ 30 秒内自动拉起 |
| 单次 GC 暂停 | ✅ G1/ZGC < 10ms |
| 网络抖动 | ✅ 重试 + 超时 |
| 磁盘写满 | ⚠️ 需要监控 + 告警 + 人工介入 |
| 整机宕机 | ❌ 必须多机 |
| 机房断电 | ❌ 必须多机房 |
| 地震 / 火灾 | ❌ 必须异地容灾 |

> **结论**:单机可以做"应用级高可用"(99.9% ~ 99.95%),但要做"基础设施级高可用"(99.99%+)必须多机。
>
> **对 spring-watch 的意义**:作为"中小企业 / 个人开发者"的轻量监控平台,99.9% 单机高可用 + 一份灾备快照,已经覆盖 80% 真实场景。盲目堆集群反而违背"5 分钟拉起"的定位。

### 2.4 一个判断决策树

```
需求:平台挂了 5 分钟能不能接受?
├── 能(非关键监控) → 单机 + 自动重启 + 本地降级 ✅ 足够
└── 不能(关键监控)
    ├── 5 分钟内必须恢复 → 同城双机 + Keepalived
    └── 1 分钟内必须恢复 → 双机房 + 数据库主从 + 流量调度
```

spring-watch v1.6 的部署形态已经 **明确选择了"单机 + 自动重启 + 本地降级"这条路径**,并通过 `白皮书 0.5` 把"为什么不集群化"的理由写清楚了。

## 三、这个项目要做到哪些才算"搞科研"?

> 下面区分三个层级:**工程实现** < **技术报告** < **科研论文**。
> spring-watch 目前完成到第二层,要打到第三层,需要补足以下工作。

### 3.1 科研产出的判定标准

| 维度 | 工程实现 | 技术报告 | 科研论文 |
|------|----------|----------|----------|
| **是否回答一个科学问题** | ❌ | ⚠️(隐含) | ✅ 必须明确 |
| **是否可证伪的假设** | ❌ | ⚠️ | ✅ 必有 |
| **是否与现有 SOTA 对比** | ❌ | ⚠️ | ✅ 必有基线 |
| **是否有新算法/新协议/新模型** | ❌ | ⚠️(渐进) | ✅ 必有 novelty |
| **是否有可复现的实验** | ❌ | ⚠️ | ✅ 必公开 |
| **是否有形式化证明或量化建模** | ❌ | ❌ | ✅ 强加分项 |

### 3.2 围绕 spring-watch 可挖的**科学问题**(候选)

> 每一个都是一个独立的科研选题,选 1~2 个深挖即可。

#### Q1:**拉取模型在单实例下的容量上界是多少?**

- **现状**:业界普遍认为"Pull 模型扛不住大规模,要上 Push"(Prometheus 官方文档也有类似论断)
- **假设 H1**:在保证告警 p99 < 30s 的前提下,**单实例的拉取容量上界由 IO + 调度算法共同决定**,而不仅仅是 CPU
- **可证伪**:能跑出 1000 目标 / 5000 目标 / 10000 目标的拐点曲线
- **实验设计**:
  - 自建 `benchmark-harness`(可参数化目标数、采集频率、Payload 大小)
  - 对比 baseline:Prometheus(单实例)+ node_exporter
  - 指标:采集时延 p50/p99、丢点率、CPU、内存、告警延迟
- **预期 novelty**:**"反例证明"** —— 证明在中等规模(< 2000 目标)下,精心设计的拉取模型可以反超业界认知

#### Q2:**JVM 单实例 + 本地缓存能否替代 Redis 承接监控状态?**

- **现状**:几乎所有生产监控都假设"状态必须外置"
- **假设 H2**:**状态可重建 + 启动期反演** 的设计,在监控类业务(读多写少、状态可丢失)中可以替代分布式缓存
- **实验设计**:
  - 用 Caffeine + 启动期 InfluxDB 反演,实现 `AlertStateStore` / `LogDedupService` / `LogAnomalyDetector`
  - 量化指标:启动时间、内存占用、查询 p99、命中率、状态丢失率
  - 对比 baseline:Redis Cluster(3 主 3 从)
- **预期 novelty**:**"轻量级状态机"的形式化模型** —— 哪些状态必须持久、哪些可重建、哪些可丢弃

#### Q3:**告警状态机在分布式降级场景下的有界性证明**

- **现状**:告警状态机常用但很少有形式化保证
- **假设 H3**:在"采集-评估-通知"三段解耦 + 状态机持久化的前提下,**告警评估在有限时间内必然收敛**
- **方法**:
  - 给出状态机定义(状态集、转移条件、有界性约束)
  - 用 TLA+/Alloy 做形式化建模,证明不变量(invariant)
  - 故障注入:Kafka 阻塞、InfluxDB 抖动、邮件服务器宕机
- **预期 novelty**:**第一个面向"监控系统自身"的告警有界性证明**

#### Q4:**自监控的"自指"问题 —— 监控系统如何监控自己而不陷入循环?**

- **现象**:spring-watch 用 InfluxDB 存自监控数据,但 InfluxDB 自己也可能挂
- **科学问题**:当被观测系统的**观测通道本身成为单点**时,如何构造"无自指"的观测?
- **方法**:
  - 定义观测层级 L0(被监控目标)/ L1(spring-watch 平台)/ L2(L1 的观测)…… 是否有上限?
  - 给出"自指打破"的形式条件(类比哥德尔不完备定理的工程版)
  - 工程上提出 2 层 + 旁路心跳的方案
- **预期 novelty**:**监控系统的层级模型 + 不可观测点定理**

#### Q5:**拉取调度算法在大规模目标下的收敛性**

- **现状**:spring-watch 用了简单的定时拉取 + 分桶,但没有理论保证
- **假设 H5**:在目标数量 N、采集频率 f、网络抖动 σ 已知时,**存在最优调度策略使得采集时延方差最小**
- **方法**:
  - 把调度问题建模为 M/G/k 排队模型
  - 给出调度策略:均匀调度 / 抖动感知调度 / 优先级调度
  - 用仿真 + 实测验证
- **预期 novelty**:**拉取调度的排队论分析 + 启发式算法**

| 阶段 | 产出物 | 工作量(估) | 已有? |

### 3.5 一句话总结

> **把 spring-watch 从"一个能跑的监控项目"升级为"一篇能发的论文",关键不在功能多不多,而在"敢不敢提出一个可被证伪的论断,并用数据和形式化方法去验证它"。**
>
> 工程上它已经走通了"单机 + 拉取 + 本地缓存"这条少有人走的路;科研上,这本身就是**一个有价值的反例证据**,只要补上 benchmark + 形式化 + 对比实验,完全可以支撑一篇系统/性能类的会议论文。
