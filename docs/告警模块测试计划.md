## 三、E2E 端到端测试场景(需 spring-watch 完整运行)

### 场景 1: 简单阈值触发(单条件 + 默认 level)

**步骤**:
1. 启动 spring-watch
2. 确认 mock-test 在 `localhost:8081` 运行
3. 确认 `monitor_app` 表有 appid=1, endpoint=`http://localhost:8081`, metrics_port=9464
4. 灌入场景 1 规则
5. 等待 15s(采集周期),触发 metric 流入

**预期日志**:
```
[Alerter] 批次处理完成 - total=500, submitted=500, failed=0
[Alerter] 条件首次满足, 进入PENDING - ruleId=N, appid=1, ...
[Alerter] 告警触发 - ruleId=N, appid=1, ...
[Alerter] 告警历史持久化 - historyId=N, notifyResult={"status":"ok",...}
[Alerter] 邮件发送成功 - to=test@x.com, type=firing
```

**预期邮件**(查看 MailHog `http://localhost:8025`):
- 主题: `[WARNING] myapp - jvm_memory_used_bytes = 12345.00 (rule: TEST_SIMPLE)`
- 正文: 包含应用名、appid、指标值、规则名

**验证**:
- [ ] `alert_history` 表新增 1 条记录,`alert_level='warning'`,`resolved_at=null`
- [ ] `alert:state:ruleId:appid` Redis Hash 状态=`FIRING`
- [ ] 邮件已收到,主题前缀正确

### 场景 2: 自定义 level(CRITICAL / WARNING / INFO)

**步骤**:
1. 灌入场景 2 (level=critical) 和场景 3 (level=info) 规则
2. 等待触发

**预期**:
- TEST_CRITICAL 邮件主题: `[CRITICAL] ...`
- TEST_INFO 邮件主题: `[INFO] ...`
- `alert_history.alert_level` 分别为 'critical' 和 'info'

**验证**:
- [ ] 3 个 level 各有 1 条历史记录,level 字段正确
- [ ] 邮件主题前缀正确

### 场景 3: JEXL 多条件(AND/OR/tags)

**步骤**:
1. 灌入场景 4 (AND), 5 (OR), 6 (tags) 规则
2. 等待触发

**预期**:
- TEST_JEXL_AND: 只在 `80 < value < 95` 区间触发
- TEST_JEXL_OR: 在 `metric=="jvm_heap"` 或 `metric=="gc_pause"` 时触发
- TEST_JEXL_TAGS: 只在 `method=="GET" && value>500` 时触发

**验证**:
- [ ] 各规则只在对应条件满足时进 alert_history
- [ ] 多 metric 进入 Kafka 后被正确分流

### 场景 4: times 触发次数控制

**步骤**:
1. 灌入场景 7 规则 (times=3, duration=0)
2. 持续 1 分钟观察

**预期日志**:
```
[Alerter] 条件首次满足, 进入PENDING - ruleId=N
[Alerter] 触发计数中 - ruleId=N, count=1/3
[Alerter] 触发计数中 - ruleId=N, count=2/3
[Alerter] 告警触发 - ruleId=N  (count 达到 3)
[Alerter] 邮件发送成功
```

**Redis 验证**:
```bash
redis-cli HGETALL alert:state:N:1
# 期望: state=FIRING, firstBreachAt=..., lastFiredAt=...
# 期望: triggerCount 已被 clearTriggerCount 清空
```

**验证**:
- [ ] 1 分钟后才触发(而不是立即)
- [ ] 期间 Redis `triggerCount` 字段从 1 涨到 3
- [ ] 触发后 `triggerCount` 字段被清空

### 场景 5: 自定义 template 渲染

**步骤**:
1. 灌入场景 8 规则 (template = `[{{level}}] {{app}}.{{metric}}={{value}} > {{threshold}} 规则={{rule}}`)
2. 等待触发
3. 查看邮件正文

**预期邮件正文**:
```
[CRITICAL] mock-test.jvm_memory_used_bytes=12345.00 > 0.00 规则=TEST_TEMPLATE
```

**验证**:
- [ ] 8 个占位符全部正确替换
- [ ] 没有未替换的 `{{xxx}}` 残留

### 场景 6: 沙箱拒绝(防 RCE)

**步骤**:
1. 灌入场景 9 规则
2. 持续观察 5 分钟

**预期日志**:
```
[Alerter] JEXL 表达式执行失败 - expr=value.toString().length() > 0, error=method call error
```

**预期**:
- 该规则**永不触发**任何告警
- 进程**无任何异常**抛出
- `alert_history` 表**不增加**该规则的记录

**验证**:
- [ ] 5 分钟后 `alert_history` 该规则记录数 = 0
- [ ] 日志持续打印 WARN(说明 JEXL 在拒绝而非静默吞掉)

### 场景 7: disabled 规则不评估

**步骤**:
1. 灌入场景 10 规则 (status=disabled)
2. 持续观察 5 分钟

**预期**:
- 该规则**不参与评估**
- `alert_history` 表**不增加**该规则的记录
- 日志**无任何** `ruleId=N` (disabled 的 id) 出现

**验证**:
- [ ] 5 分钟后 `alert_history` 该规则记录数 = 0
- [ ] 没有任何 `ruleId` 引用该规则的 INFO/WARN 日志

### 场景 8: 告警恢复(状态机 RESOLVED)

**步骤**:
1. 让一个规则先进入 FIRING(灌入数据)
2. 停掉 mock-test(或修改 expression 让条件不再满足)
3. 等下一次 metric 进来,条件不满足

**预期日志**:
```
[Alerter] 告警恢复 - ruleId=N, appid=1
[Alerter] 告警历史标记恢复 - historyId=N, resolvedAt=...
[Alerter] 邮件发送成功 - type=resolved
```

**预期**:
- `alert_history.resolved_at` 不再为 null
- 发送了"恢复"邮件
- Redis state 回到 IDLE (因为 resolve 后 clear)

**验证**:
- [ ] 同一 historyId 的 `resolved_at` 被填充
- [ ] 恢复邮件主题是 `[RESOLVED]`
- [ ] Redis 状态被清空

### 场景 9: PENDING 期间条件恢复(不通知)

**步骤**:
1. 灌入 `duration_seconds=60` 的规则
2. 等条件满足,进入 PENDING
3. **在 60s 内**让条件不满足
4. 等下一次 metric 进来,条件不满足

**预期**:
- 状态机: IDLE → PENDING → IDLE (静默)
- **不发送**告警邮件
- **不发送**恢复邮件

**验证**:
- [ ] PENDING 中恢复,日志打印 `PENDING中恢复, 清除状态`
- [ ] 邮件数量 = 0 (无触发也无恢复)

### 场景 10: 表达式试运行(simple vs JEXL 自动分流)

**步骤**:
1. 灌入 2 条规则:
   - `expression = "jvm_memory_used_bytes > 0"` (simple 格式)
   - `expression = "value > 0 && metric == 'jvm_memory_used_bytes'"` (JEXL 格式)
2. 等待触发

**预期**:
- 两条规则都触发(自动检测格式)
- 旧规则走 simple 分支(向后兼容)
- 新规则走 JEXL 分支

**验证**:
- [ ] `alert_history` 增加 2 条
- [ ] 日志中 simple 格式走 `simpleEvaluate`,JEXL 格式走 `JexlExprEvaluator`

## 四、性能与稳定性测试

### 4.1 高频事件压测

**场景**: 在 10s 内灌入 10000 条 metric 事件

**步骤**:
```bash
# 用 kafka-console-producer 或自定义脚本
for i in $(seq 1 10000); do
  echo "{\"appid\":1,\"metricName\":\"jvm_heap\",\"value\":$((RANDOM % 100)),\"timestamp\":\"2026-06-14T12:00:00Z\"}"
done | kafka-console-producer --bootstrap-server localhost:9092 --topic monitor-metrics
```

**预期**:
- BatchAlertConsumer 接收,按 500/批处理
- AsyncAlertExecutor 8 虚拟线程评估
- **不丢事件**,不卡死

**验证指标**:
- 内存: < 1GB 增长
- CPU: 突发但很快回落
- 日志: 无 ERROR

### 4.2 Redis 不可用降级

**步骤**:
1. 启动应用,正常跑
2. `docker stop redis`
3. 灌入数据

**预期**:
- AlertStateStore 操作抛异常
- AlertEngine catch 住,记录日志
- **不阻塞**消费者循环
- Redis 恢复后状态机自愈(因为有 IDLE 兜底)

**验证**:
- [ ] 消费者持续运行
- [ ] 日志有 ERROR 但不致命
- [ ] Redis 恢复后,新告警正常工作

### 4.3 Kafka 不可用降级

**步骤**:
1. 启动应用
2. `docker stop kafka`
3. 等待 5s
4. `docker start kafka`

**预期**:
- KafkaFallbackQueue 接收失败的 send
- 5s 后后台重投
- 恢复后队列清空

**验证**:
- [ ] 日志显示 `[spring-watch: Kafka发送失败, 转入降级队列 - ...]`
- [ ] 日志显示 `[spring-watch: KafkaFallbackQueue drain - sent=N, ...]`
- [ ] Kafka 恢复后队列 size = 0

### 4.4 告警风暴压测(去重)

**场景**: 同一规则 1s 内触发 100 次

**预期**:
- 状态机进入 FIRING 后,**不重复**发邮件
- `alert_history` 只有 1 条新记录
- Redis state = FIRING (不会回退)

**验证**:
- [ ] 邮件数量 = 1
- [ ] `alert_history` 新增 = 1

## 六、回滚方案(任何阶段出问题)

### 6.1 回滚 P0

```sql
-- 移除 level 列
ALTER TABLE alert_rule DROP COLUMN level;
```

代码层面:AlertEngine / AlertNotifier 的改动**与旧逻辑兼容**(`level=null` 默认 warning,行为同改前)。

### 6.2 回滚 P1

```sql
-- 移除 JEXL 表达式(只能用 simple 格式的规则)
DELETE FROM alert_rule WHERE expression ~ '[&|]';  -- 含 && 或 || 的
```

代码层面:把 `JexlExprEvaluator` 的依赖从 `AlertEvaluator` 移除,恢复旧 `isBreached` 方法。

### 6.3 回滚 P2

```sql
ALTER TABLE alert_rule DROP COLUMN template;
ALTER TABLE alert_rule DROP COLUMN times;
```

代码层面:`AlertEngine.handleBreach` 改回只用 `duration_seconds` 触发,`AlertNotifier.buildBody` 改回 default 模板。
