# 系统指标可视化规划

## 一、可用指标清单(来自 mock-test `/metrics`)

| 指标名 | 类型 | 单位 | 标签 | 数据特征(示例) |
|---|---|---|---|---|
| `system_disk_io_bytes_total` | counter | bytes | `device`、`direction` | 5 块盘 × 读/写,数量级差异大(0 ~ 1.1E10) |
| `system_disk_operations_total` | counter | ops | `device`、`direction` | 5 块盘 × 读/写,读多写少或反之 |
| `system_memory_usage_bytes` | gauge | bytes | `state` | free/used 二态,适合比例展示 |
| `system_memory_utilization` | gauge | ratio | `state` | free/used(0~1),本质同 bytes,二选一 |
| `system_network_errors_total` | counter | errs | `device`、`direction` | 多数时间为 0,告警语义强 |
| `system_network_io_bytes_total` | counter | bytes | `device`、`direction` | 接收/发送量级对比 |
| `system_network_packets_total` | counter | pkts | `device`、`direction` | 接收/发送包量 |

## 三、推荐图表方案

### 1. CPU 模块
- **CPU 时序折线图**(已存在):user / system 两条折线
- **新增 CPU 利用率堆叠面积图**:可选 `system_cpu_utilization`,user / system / iowait 三段堆叠

### 2. 内存模块
- **内存时序堆叠面积图**(已存在,used/free)
- **新增 内存使用率仪表盘**:以 `system_memory_utilization{state="used"}` 作大圆环/进度条,百分比直读
- **新增 内存分布饼图**:free / used 两块饼,体积小、读数直观

### 3. 磁盘模块
| 图表 | 类型 | 维度 | 数据来源 |
|---|---|---|---|
| 磁盘 IO 字节(分方向) | **分组柱状图** | X=device,系列=read/write | `system_disk_io_bytes_total`,按 device 分组后内部按 direction 拆分 |
| 磁盘 IOPS(分方向) | **分组柱状图** | X=device,系列=read/write | `system_disk_operations_total`,同上 |
| 磁盘 IO 字节时序 | **堆叠面积图** | 系列=各 device | `system_disk_io_bytes_total` series(取 top N 设备) |
| 磁盘 IOPS 时序 | **折线图** | 系列=各 device | `system_disk_operations_total` series |
| 磁盘使用率(可选) | **水平柱状图** | Y=device,X=used% | `system_filesystem_usage` 类(待确认是否采集) |

### 4. 网络模块
| 图表 | 类型 | 维度 | 数据来源 |
|---|---|---|---|
| 网络 IO 字节(分方向) | **分组柱状图** | X=device,系列=receive/transmit | `system_network_io_bytes_total` |
| 网络包数(分方向) | **分组柱状图** | X=device,系列=receive/transmit | `system_network_packets_total` |

> 计数器(counter)在单点查询时只能拿到"累计值",柱状图展示的是"历史总量";时序图通过 `rate()` 才能展示真实速率。当前后端 `querySeries` 已支持 `agg=rate`,规划时序图统一传 `agg=rate`,柱状图用 `latest` 取最新累计值。

# HTTP 指标可视化规划

## 一、可用指标与数据样本

指标:`http_server_request_duration_seconds`(histogram),标签维度:
- `http_request_method` (GET/POST/...)
- `http_response_status_code` (200/404/500/...)
- `http_route` (路由模板,如 `/api/orders/{id}`)
- `url_scheme` (http)
- `network_protocol_version`
- `otel_scope_*`(固定,无需展示)

样本数据(从 metrics.txt 抓):
- 7 个具名路由 + 1 个 `/**` 兜底(404)
- 累计请求:`/api/ping` 953(最高)、`/api/users/{id}` 352、`/api/products/{id}` 235、`/api/products` 224、`/**` 210、`/api/users` 124、`/api/users/{id}/stats` 120、`/api/orders/{id}` 103

## 二、关键约束:数据懒加载 + 图表动态生成

路由列表是动态的(取决于应用暴露了哪些接口),不能写死在 HTML 中。
流程:
1. **首查路由清单**:`grouped` 用 `groupBy=http_route` 拉一次,得到所有 (route, method, status) 组合
2. **动态创建面板**:遍历清单,为每个组合生成 `.chart-panel` 节点,插入到 `#http-route-panels` 容器
3. **延迟加载图表数据**:每个面板创建后,异步拉该路由的 histogram-quantile + QPS series,渲染图表
4. **后续优化(可选)**:用 IntersectionObserver 实现"滚到视区再加载",避免一次 N 个并发请求

## 三、布局设计

```
┌─ 顶部:HTTP 概览(原有 4 卡)───────────────────────────────────────┐
│ 总请求数 │ 不同路由数 │ 不同状态码数 │  错误占比(4xx+5xx)         │
└──────────────────────────────────────────────────────────────────┘

┌─ 总体区(2 行)──────────────────────────────────────────────────┐
│ 行1: 状态码分布饼(200/404/500) │ HTTP 方法分布饼(GET/POST/...)  │
│ 行2: Top10 路由按请求数柱        │ Top10 路由按 P99 柱           │
│ 行3: 总 QPS 趋势(双轴:成功/失败)│ 总体 P50/P95/P99(全站聚合)   │
└──────────────────────────────────────────────────────────────────┘

┌─ 动态区 ────────────────────────────────────────────────────────┐
│ <h3>按接口(动态生成)</h3>                                      │
│ <div id="http-route-panels">                                   │
│   [动态创建 1:N 个 panel]                                       │
│   模板:                                                         │
│   ┌──────────────────────────────────────────────────────────┐  │
│   │ GET /api/orders/{id} 200                                 │  │
│   │ 总 103 次 │ 平均 X ms │ 错误 0                            │  │
│   │ [P50/P95/P99 折线]  [QPS 折线]                           │  │
│   └──────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

## 四、每路由面板数据

| 字段 | 数据来源 | API |
|---|---|---|
| 总次数 | `http_server_request_duration_seconds_count` latest | `/api/metrics/latest` |
| 平均耗时 | `_sum / _count`(同次拉两个 latest) | `/api/metrics/latest` × 2 |
| 错误次数 | `_count` 过滤 `http_response_status_code=~"4..\|5.."` | grouped |
| P50/P95/P99 | histogram-quantile 过滤 `http_route=...&http_request_method=...&http_response_status_code=...` | `/api/metrics/histogram-quantile` |
| QPS 趋势 | `_count` series 用 `agg=rate` | `/api/metrics/series` |

> 同样的 (route, method, status) 组合作为不同面板(同路由不同状态码分开画)。这样 404 路由 `/**` 的"找不到"问题能直接看见。

## 五、落地

1. **后端**:无需大改,`grouped` 多维分组 + `histogram-quantile` tag 过滤 + `series agg=rate` 都已具备
2. **前端**:
   - HTML:在 HTTP 区底部加 `<div id="http-route-panels"></div>`,移除"慢请求 TopN"那个静态卡(用动态版替代)
   - JS:
     - `renderHttp()`:先调 grouped 拉清单,再循环 `createRoutePanel(routeKey)` 创建 DOM
     - `createRoutePanel()`:构建 panel 节点 → 插入容器 → 异步 `loadRouteCharts(panelEl, routeKey)` 拉数据 + 渲染
     - `loadRouteCharts()`:并行拉 4 个 API,渲染 2 个图(分位 + QPS)+ 3 个文本(总次/平均/错误)
3. **优先级**:
   - P0:动态面板 + 分位 + QPS(主要价值)
   - P1:总 QPS 趋势 + 错误占比卡
   - P2:IntersectionObserver 懒加载优化

## 六、待确认

- 默认要不要给 `/**` 404 单独画一个面板?(建议画,并标红警示)
- 路由数量很多时(几百个)是否需要折叠/分页?(P2 优化,先做全量)

### 2. 内存区
| 图表 | 类型 | 数据来源 |
|---|---|---|
| 堆内存使用堆叠(已) | 时序堆叠面积 | `jvm_memory_used_bytes` × heap |
| 非堆内存使用堆叠(已) | 时序堆叠面积 | `jvm_memory_used_bytes` × non_heap |
| **各 pool 已用 vs 上限** | 横向条形,used% 着色 | `jvm_memory_used_bytes / jvm_memory_limit_bytes` 单点 |
| **堆 committed 趋势** | 时序线,各 pool 拆分 | `jvm_memory_committed_bytes` × heap |
| **GC 后剩余内存** | 柱图(各 pool) | `jvm_memory_used_after_last_gc_bytes` |

### 3. GC 区
| 图表 | 类型 | 数据来源 |
|---|---|---|
| **GC 累计次数(替换)** | 分组柱 `name × action` | `jvm_gc_duration_seconds_count` grouped by `jvm_gc_name,jvm_gc_action` |
| **GC 平均耗时** | 柱图,按 `gc_name` | `_sum / _count` 算 mean,按 name 聚合 |
| **GC P50/P95/P99** | 多线时序图,按 `gc_name` | histogram-quantile,需 API 支持 tag 过滤 |
| **GC P99 大值告警** | 在 P99 折线图上对 > 阈值(如 200ms)上红 | 同上 |

### 4. 线程/类加载 区
| 图表 | 类型 | 数据来源 |
|---|---|---|

| **类加载速率** | 时序线,`agg=rate` | `jvm_class_loaded_total` |
| **类卸载速率** | 时序线,`agg=rate` | `jvm_class_unloaded_total` |

## 四、布局建议(替换现有 4 卡 + 2 行)

```
┌─ 顶部 6 个数字卡 ────────────────────────────────────────────────┐
│ CPU 使用率 │ 当前加载类 │ 累计已加载类 │ 累计已卸载类 │           │
│ 总线程数   │ JVM CPU 时间累计(秒)                               │
└──────────────────────────────────────────────────────────────────┘

┌─ 内存区(2 行)───────────────────────────────────────────────────┐
│ 行1: 堆内存使用堆叠(已有)  │ 非堆内存使用堆叠(已有)             │
│ 行2: 各 pool 已用 vs 上限   │ 堆 committed 时序                 │
└──────────────────────────────────────────────────────────────────┘

┌─ GC 区(2 行)────────────────────────────────────────────────────┐
│ 行1: GC 累计次数(name×action)  │ GC 平均耗时(name)              │

┌─ 线程/类加载 区 ─────────────────────────────────────────────────┐

└──────────────────────────────────────────────────────────────────┘
```

## 五、落地优先级

1. **后端**:扩展 `histogram-quantile` API 支持 tag 过滤(必做,GC P95/P99 多线依赖)
2. **前端**:
   - P0:GC 累计次数(name×action)、GC 平均耗时、类加载/卸载速率、线程 daemon×state、内存各 pool 占比
   - P1:GC P50/P95/P99 多线、堆 committed 趋势、GC 后内存
   - P2:数字卡扩展(累计已卸载类、JVM CPU 时间累计、总线程数)

## 六、待确认

- `jvm_memory_limit_bytes` 在示例中只有 5 个 pool 有值(G1 Eden、Survivor 无 limit 字段,因 G1 不预先划分)——占比图需在缺失 limit 的 pool 上做"无上限"提示
- `jvm_class_loaded_total` 是 counter,速率图需用 `agg=rate` 拉取,InfluxDB 该指标若已写过 rate 是否会被重置?需测试确认
