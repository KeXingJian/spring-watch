## 一、报告背景与目标

### 1.1 背景

spring-watch 当前已落地:
- ✅ Kafka 批量生产消费架构
- ✅ 告警状态机(IDLE→PENDING→FIRING→RESOLVED)
- ✅ 邮件通知(JavaMailSender)
- ✅ 简单阈值规则(正则解析 `metricName op value`)

但已发现**3 个天花板**,开始制约业务:
1. **规则格式太弱** — 正则只支持 `metricName op value`,写不了多条件
2. **没有严重等级选择** — 写死在引擎里 critical/warning
3. **没有表达式试运行工具** — 写错规则只能等真数据触发才知道

HertzBeat 完整跑通了这套体系(7 年积累,1.7 万 star),是合适的参考。

### 1.2 目标

| 目标 | 衡量标准 |
|---|---|
| 短期打通:跟 HertzBeat 看齐的规则能力 | 表达式能写多条件 / 用户能选等级 / 能试运行 |
| 中期演进:工业级告警 | 引入 JEXL / 引入 times 触发次数 / 模板占位符 |
| 长期目标 | 规则模板市场 + AI 自动生成规则 |

## 二、HertzBeat 告警规则体系全景

### 2.1 设计哲学:双引擎 + 双模式

HertzBeat 把告警规则拆成**两种触发模式**,用**两套完全不同的表达式引擎**:

```
┌─────────────────────────────────────────────────────────────┐
│                    AlertDefine (规则)                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   type=realtime_* ──► JEXL 表达式(单事件评估)                │
│   │  - realtime_metric                                       │
│   │  - realtime_log                                          │
│   │                                                          │
│   type=periodic_* ──► ANTLR DSL(周期评估,支持 PromQL+SQL)    │
│      - periodic_metric                                       │
│      - periodic_log                                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

| 模式 | 触发时机 | 引擎 | 适用场景 |
|---|---|---|---|
| `realtime_metric` | 每条 metric 来 | JEXL | 单事件阈值 (`heap > 80%`) |
| `realtime_log` | 每条 log 来 | JEXL | 日志模式匹配 (`severity == ERROR`) |
| `periodic_metric` | 每 N 秒(period) | ANTLR DSL | 聚合/速率 (`sum(rate(x[5m])) > 0.8`) |
| `periodic_log` | 每 N 秒 | ANTLR DSL | 日志聚合查询 |

**为什么拆?** 因为 "heap > 80%" 和 "5min 错误率 > 1%" 是**两种完全不同的判断**,前者一条 metric 就够,后者需要时间窗口。

### 2.2 数据模型

#### 2.2.1 `hzb_alert_define` 表

```sql
CREATE TABLE hzb_alert_define (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    name        VARCHAR(100)  NOT NULL,             -- 规则名(唯一)
    type        VARCHAR(255),                        -- realtime_metric | periodic_metric | realtime_log | periodic_log
    expr        VARCHAR(2048),                       -- ★ 关键:表达式
    period      INTEGER,                             -- 周期/窗口(秒)
    times       INTEGER,                             -- ★ 触发 N 次才升级
    labels      VARCHAR(2048) (JSON),                -- {"severity":"critical", "team":"ops"}
    annotations VARCHAR(4096) (JSON),                -- {"summary":"..."}
    template    VARCHAR(2048),                       -- 消息模板 "Instance {{$labels.instance}} CPU={{$value}}%"
    datasource  VARCHAR(100),                        -- PROMETHEUS | sql (periodic 专用)
    enable      BOOLEAN       DEFAULT TRUE,
    creator     VARCHAR(50),
    gmt_create  TIMESTAMP,
    gmt_update  TIMESTAMP
);
```

#### 2.2.2 关键设计取舍

| 设计点 | 选型 | 理由 |
|---|---|---|
| 与 app 的关系 | expr 里写 `equals(__app__, "kafka")` | 一个规则可绑多 app,比外键灵活 |
| 严重等级 | 塞进 `labels.severity` JSON | 避免 schema 改字段 |
| 触发控制 | `times` 次数 + `period` 窗口 | 比 `duration` 精细 |
| 消息格式 | `template` 字段,支持 `{{$value}}` 占位符 | 模板化,前端可定制 |
| 数据源 | `datasource` 字段(PROMETHEUS/sql) | periodic 模式需要指定查询目标 |

#### 2.2.3 `hzb_alert_define_monitor_bind` 多对多绑定表

```sql
CREATE TABLE hzb_alert_define_monitor_bind (
    id              BIGINT PRIMARY KEY,
    alert_define_id BIGINT,
    monitor_id      BIGINT  -- 具体实例 ID
);
```

**双模式绑定**:
- **自动**:规则 expr 里 `equals(__app__, "kafka")` 匹配该应用下所有实例
- **显式**:绑具体 monitor ID,只对那几个实例生效

### 2.3 JEXL 表达式引擎(realtime_*)

#### 2.3.1 核心调用

```java
// JexlExprCalculator.java
public boolean execAlertExpression(Map<String, Object> fieldValueMap, String expr, boolean ignoreJexlException) {
    Boolean match = (Boolean) JexlExpressionRunner.evaluate(expression, fieldValueMap);
    return match != null && match;
}
```

#### 2.3.2 注入的变量(在 `MetricsRealTimeAlertCalculator.java:179-193` 构造 context)

```java
Map<String, Object> context = new HashMap<>();
context.put("__app__",           app);           // 应用名
context.put("__metrics__",       metrics);       // 指标名
context.put("__instance__",      instanceId);

```

#### 2.3.3 支持的操作符和函数

| 类别 | 语法 |
|---|---|
| 比较 | `>`, `>=`, `<`, `<=`, `==`, `!=` |
| 逻辑 | `&&`, `\|\|`, `!`(也支持 `and`/`or`/`not`) |
| 字符串 | `equals(a, b)`, `!equals(a, b)` |
| 包含 | `contains(a, b)`, `!contains(a, b)`(大小写不敏感) |
| 正则 | `matches(s, regex)`, `!matches(s, regex)` |
| 存在 | `exists(x)`, `!exists(x)` |
| 时间 | `now()` 返回当前毫秒 |
| JSON | `json("{\"a\":1}")` 解析为 Map |

#### 2.3.4 沙箱化(关键安全)

```java
// JexlExpressionRunner.java
features.annotation(false).loops(false).pragma(false)
        .methodCall(false).lambda(false).newInstance(false).register(false);
```

**禁用**:循环 / lambda / 方法调用 / newInstance / 注解 / register。

### 2.4 ANTLR DSL 表达式引擎(periodic_*)

完整 grammar 在 `hertzbeat-alerter/src/main/resources/expr/AlertExpression.g4`:

```
expr
    : LPAREN expr RPAREN                                  # ParenExpr
    | left=expr op=(GE|LE|GT|LT|EQ|NE) BOOL? right=expr    # ComparisonExpr
    | left=expr AND right=expr                              # AndExpr
    | left=expr UNLESS right=expr                           # UnlessExpr
    | left=expr OR right=expr                               # OrExpr
    | promql                                                # PromqlExpr
    | selectSql                                             # SqlExpr
    | number                                                # LiteralExpr
    | SQL_FUNCTION LPAREN string RPAREN                     # SqlCallExpr
    | PROMQL_FUNCTION LPAREN string RPAREN                  # PromqlCallExpr
    ;
```

**支持的 PromQL 函数**: `sum`, `avg`, `count`, `min`, `max`, `stddev`, `variance`, `topk`, `bottomk`, `quantile`, `rate`, `increase`, `histogram_quantile`

**支持的 SQL**: `SELECT ... FROM ... WHERE ... GROUP BY ... HAVING ... ORDER BY ... LIMIT ...`

**支持的 SQL 算子**: `LIKE`, `NOT LIKE`, `IN`, `NOT IN`, `IS`, `IS NOT`, `BETWEEN`

### 2.5 严重等级(在 labels 里)

```java
// CommonConstants.java
ALERT_SEVERITY_EMERGENCY = "emergency"  // 用户可选
ALERT_SEVERITY_CRITICAL  = "critical"   // 用户可选
ALERT_SEVERITY_WARNING   = "warning"    // 用户可选
ALERT_SEVERITY_INFO      = "info"       // 内部用
```

**存储**:塞进 `labels` JSON 字段:
```json
{"severity": "critical", "team": "ops", "alert_mode": "group"}
```

### 2.6 状态机(三状态)

```java
ALERT_STATUS_PENDING  = "pending"   // 触发中,等待 times 达到
ALERT_STATUS_FIRING   = "firing"    // 已升级,通知已发
ALERT_STATUS_RESOLVED = "resolved"  // 恢复
```

**`times` 字段是核心**:
- 不是 `duration`(持续时长),而是**触发次数**
- `times=1` = 单次触发 = 跟 spring-watch 现在的 duration=0 类似
- `times=3, period=300` = "5 分钟内匹配 3 次才告警"(防抖动)
- 比 duration 更精确:既能防单点抖动,也能防短时爆发

### 2.7 真实规则例子(从代码里扒的)

```java
// 1. 高 CPU 周期告警(PromQL)
AlertDefine.builder()
    .name("high_cpu_usage")
    .type("periodic")
    .expr("usage>90")
    .times(1).period(300)
    .labels(Map.of("team", "test-team", "priority", "P1"))
    .template("High CPU usage (${__value__}%)")
    .datasource("PROMETHEUS")
    .build()

// 2. 日志 ERROR 实时告警(JEXL)
AlertDefine.builder()
    .name("test_log_alert")
    .type("realtime_log")
    .expr("log.severityText == 'ERROR'")
    .template("Alert: {{log.body}}")
    .labels(Map.of("severity", "critical"))
    .build()

// 3. PromQL 聚合(从 i18n 提示语)
"sum(rate(process_cpu_seconds_total[5m])) > 0.8"

// 4. 多 app + 多条件 + 嵌套
"equals(__app__,\"openai\") && equals(__metrics__,\"credit_grants\") " +
"&& total_used > 123 and total_granted > 333 " +
"and (total_granted > 3444 and total_paid_available < 5556)"
```

## 三、spring-watch 告警现状审计

### 3.1 已落地能力

| 模块 | 能力 |
|---|---|
| `AlertEvaluator.isBreached(rule, event)` | 单条 metric + 6 个比较操作符 + 阈值 |
| `AlertStateStore` | Redis Hash 存状态机,TTL 24h |
| `AlertRuleCache` | 30s 刷新 + Controller 失效(已移除 controller) |
| `AlertEngine` | IDLE→PENDING→FIRING→RESOLVED 完整流转 |
| `AlertNotifier` | 邮件(SMTP),解析 `notifyChannels` JSON |
| `BatchAlertConsumer` | Kafka 批量消费 + 异步评估线程池 |
| `AsyncAlertExecutor` | 8 虚拟线程,异步不阻塞消费 |

### 3.2 数据库 schema

```sql
CREATE TABLE alert_rule (
    id BIGSERIAL PRIMARY KEY,
    app_id BIGINT REFERENCES monitor_app(id),
    rule_name VARCHAR(256),
    rule_type VARCHAR(32),          -- 当前只支持 "metric"
    expression VARCHAR(512),        -- 简单正则
    threshold_value DOUBLE PRECISION,
    duration_seconds INTEGER DEFAULT 60,
    notify_channels JSONB,          -- {"email":"ops@x.com"}
    status VARCHAR(16) DEFAULT 'enabled',  -- enabled/disabled
    created_at TIMESTAMPTZ
);
```

### 3.3 已落地的优势

1. **外键绑 app**(`app_id`)—— 直观,符合 spring-watch 数据规模(单租户,几十~几百 app)
2. **明确的状态机** —— 比 HertzBeat 三状态更细致(IDLE/PENDING/FIRING/RESOLVED)
3. **Kafka 异步链路** —— 解耦消费与告警评估
4. **降级队列** —— Kafka 失败本地兜底,HertzBeat 也没有

## 四、差距分析与风险评估

### 4.1 功能差距

| 维度 | HertzBeat | spring-watch | 风险 |
|---|---|---|---|
| **表达式能力** | JEXL 多条件 + 函数 | 单条正则 | 🔴 高 —— 用户写不出"heap>80 && gc>5s"这种 |
| **严重等级** | 用户可选 3 档 | 引擎硬编码 | 🟡 中 —— 业务需要 critical/warning 选择权 |
| **多条件组合** | `&& / \|\| / ()` | ❌ | 🔴 高 —— 实际场景 80% 是组合条件 |
| **聚合函数** | `rate / avg / quantile` | ❌ | 🟡 中 —— "5min 错误率"做不到 |
| **times 触发次数** | ✅ | ❌(只有 duration) | 🟡 中 —— duration=60 配 times=1 等价,但不够灵活 |
| **表达式试运行** | `/preview` 端点 | ❌ | 🟢 低 —— 调试工具,优化项 |
| **消息模板** | `{{$value}}` 占位符 | 硬编码文本 | 🟢 低 |
| **导入导出** | JSON/YAML/EXCEL | ❌ | 🟢 低 |
| **多数据源** | PROMETHEUS/SQL | 单一(Kafka 流) | ✅ 不需要 |
| **AI 生成规则** | 有 | ❌ | 🟢 低(长期) |

### 4.2 风险评估

**🔴 高优先级(业务受阻)**:
- 多条件写不了 → 用户被迫写 N 条规则,告警洪水 / 漏报
- 严重等级不可选 → 业务侧无法分级响应

**🟡 中优先级(竞争力)**:
- 聚合函数缺失 → 5min 错误率等常见需求做不了
- times 缺失 → 抖动场景只能靠 duration 凑

**🟢 低优先级(锦上添花)**:
- 表达式试运行 / 模板 / 导入导出 / AI —— 体验优化,不影响主流程

### 4.3 复杂度评估

引入 JEXL 的成本:
- 依赖:`commons-jexl3` 约 500KB
- 沙箱配置:1 个文件
- 表达式解析:替换 `AlertEvaluator.isBreached` 的 30 行 → 改成调用 JEXL
- 测试:覆盖单条件 / 多条件 / 沙箱边界
- **预计 1 人天**

引入 times + template + 试运行端点:预计 **1 人天**。

### 5.1 P0(本周,1 天)

**目标**:打通用户最迫切的"等级选择"和"调试能力",不改 schema 主体。

#### 改动清单
- [ ] `AlertRule` 实体加 `level` 字段(V6 migration,`VARCHAR(16) DEFAULT 'warning'`)
- [ ] `AlertRuleCache` 加 `level` 字段透传
- [ ] `AlertEngine.determineLevel()` 改用 `rule.getLevel()` 而非计算
- [ ] `AlertNotifier` 邮件主题前缀按 level 区分(`[CRITICAL]` / `[WARNING]`)
- [ ] 新增 `AlertEvaluator.testExpression(String expr, MetricEvent sample)` 工具方法
- [ ] 新增单测 `AlertEvaluatorTest`:覆盖 6 个操作符 + 边界值

#### 风险
- 低。schema 加一列 DEFAULT 'warning',向后兼容。
- `level` 字段与现在 `determineLevel` 计算结果**可能不一致**,以**新字段为准**。

#### 验证
```sql
INSERT INTO alert_rule (..., level) VALUES (..., 'critical');
```
- 触发后邮件主题应是 `[CRITICAL] ...`
- 不传 `level` 时仍是 `warning`

### 5.2 P1(下周,1-2 天)

**目标**:引入 JEXL,支持多条件。**这是最大的能力跃迁**。

#### 改动清单
- [ ] `pom.xml` 加 `org.apache.commons:commons-jexl3:3.4.0`
- [ ] 新增 `JexlExprEvaluator` 组件(替代现有 `AlertEvaluator.isBreached`)
- [ ] 配置 `JexlEngine` Bean,启用沙箱
- [ ] 改造 `AlertEvaluator`:`isBreached(rule, event)` 改为 `JexlExprEvaluator.evaluate(rule.getExpression(), event)`
- [ ] JEXL context 注入:`metricName`(事件当前指标),`value`(事件当前值),`__app__`(appid),`__metric__`
- [ ] 兼容旧规则:正则表达式走旧分支,JEXL 走新分支(用 `contains(expr, "&&")` 判断?)
- [ ] 单测:`JexlExprEvaluatorTest`,覆盖 `metric > 80 && metric < 95` 这种

#### 数据库 schema 变更(可选)
```sql
-- V7__alert_rule_expr_type.sql
ALTER TABLE alert_rule ADD COLUMN expr_type VARCHAR(16) DEFAULT 'simple';
-- 'simple' = 旧正则格式(向后兼容)
-- 'jexl'   = 新 JEXL 表达式
```

#### 新规则示例(给用户参考)
```sql
-- P99 延迟高且错误率 > 1%
INSERT INTO alert_rule (..., expression, ...)
VALUES (..., 'value > 1000 && value < 5000', ...);

-- GC 暂停 OR 堆内存 超阈值
INSERT INTO alert_rule (..., expression, ...)
VALUES (..., 'metric == "jvm_gc_pause_seconds_max" && value > 1 || metric == "jvm_memory_used_bytes" && value > 1500000000', ...);
```

#### 风险
- **中**。JEXL 沙箱必须严格配置,否则用户能 `newInstance` 就有 RCE 风险。
- 旧规则的兼容策略:用 `__LEGACY__` 前缀区分,或迁移脚本。

### 5.3 P2(迭代,3-5 天)

**目标**:补齐 HertzBeat 的核心能力 —— times 触发次数 + template 模板。

#### 改动清单
- [ ] `AlertRule` 加 `times`(默认 1)和 `template`(默认 null)字段
- [ ] `AlertRuleCache` 透传
- [ ] `AlertEngine.handleBreach` 改用 `triggerTimes >= rule.getTimes()` 升级到 FIRING
- [ ] `AlertNotifier.notify` 模板渲染:`{{app}}` / `{{metric}}` / `{{value}}` / `{{rule}}` / `{{time}}`
- [ ] 数据库 migration V8
- [ ] 触发计数器存 Redis 或 进程内存(用现有 AlertStateStore 加字段)

#### 新规则示例
```sql
-- 5 分钟内 3 次 GC 暂停 > 1s 才告警(防单次抖动)
INSERT INTO alert_rule (..., expression, times, template)
VALUES (..., 'jvm_gc_pause_seconds_max > 1', 3,
        '【{{level}}】{{appName}} 的 {{metric}} 多次超阈值,当前={{value}}');

-- 自定义模板:钉钉友好的精简消息
INSERT INTO alert_rule (..., template)
VALUES (..., '{{metric}}={{value}} > {{threshold}} (rule: {{ruleName}})');
```

#### 风险
- **中**。`times` 需要窗口内计数,要小心状态机:同一个 (rule, app) 多次进 PENDING 怎么处理?需要明确"同一窗口内累积"还是"滑动窗口"。

## 六、每阶段落地清单

### 6.1 P0 落地清单

```java
// 1. V6__alert_rule_level.sql
ALTER TABLE alert_rule ADD COLUMN level VARCHAR(16) DEFAULT 'warning';
```

```java
        .replace("{{metric}}", event.getMetricName())
```

## 七、示例规则对照表

| 业务场景 | HertzBeat 写法 | spring-watch 当前 | 演进后 |
|---|---|---|---|
| 堆内存 > 80% | `expr="heap_memory_used > 80"` | ✅ 可写 | P0 加 `level=critical` |
| GC 暂停 > 1s | `expr="jvm_gc_pause_seconds_max > 1"` | ✅ 可写 | P0 加 `level=warning` |
| **堆内存高 AND GC 频繁** | `expr="heap > 80 && gc > 1"` | ❌ 写不了 | **P1 JEXL** |
| **5min 错误率 > 1%** | `expr="sum(rate(errors[5m])) > 0.01"` | ❌ 写不了 | P2 + ANTLR |
| **3 次 GC 暂停才告警** | `times=3, period=300` | ⚠️ 用 duration=60 凑 | **P2 times 字段** |
| **钉钉精简消息** | `template="..."` | ❌ 硬编码 | **P2 template** |
| **业务订单异常** | `expr="order_amount > 10000"` | ✅ 可写 | P0 加 `level=warning` |

## 九、报告结论

| 结论 | 说明 |
|---|---|
| **HertzBeat 的设计成熟度远超我们** | 双引擎 + 沙箱化 + 触发次数 + 模板,值得全量参考 |
| **但不要照搬** | spring-watch 是 Kafka 流(不是 PromQL/SQL),HertzBeat 的 ANTLR DSL 路径我们用不上 |
| **核心借鉴:沙箱化 JEXL** | 用最小成本解决"多条件 + 灵活函数"问题,1 人天可落地 |
| **触发次数 + 模板** | 第二阶段补齐,2-3 人天 |
| **schema 保持兼容** | 加 `level` / `times` / `template` / `expr_type` 都是非破坏性 ADD COLUMN |
| **AI / 模板市场** | 远期,等业务真到那个规模再做 |

**总投入**:P0 + P1 + P2 ≈ **5-7 人天**,获得与 HertzBeat 对齐的规则能力,且不破坏现有链路。
