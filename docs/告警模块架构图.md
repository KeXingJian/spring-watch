## 二、状态机详解

```mermaid
stateDiagram-v2
    [*] --> IDLE

    IDLE --> PENDING: 条件满足<br/>setState(PENDING, firstBreachAt=now)
    PENDING --> PENDING: 持续满足<br/>triggerCount++, elapsed++
    PENDING --> FIRING: elapsed >= durationSec<br/>或 triggerCount >= times
    PENDING --> IDLE: 条件恢复<br/>(短促抖动 clear)
    PENDING --> FIRING: 扫描器兜底<br/>(now - firstBreachAt >= durationSec)

    FIRING --> FIRING: 持续满足<br/>(防告警风暴,跳过)
    FIRING --> RESOLVED: 条件恢复<br/>setState(RESOLVED)<br/>发恢复邮件
    RESOLVED --> IDLE: 立即 clear

    IDLE --> PENDING: 重新满足
    FIRING --> [*]: 解决后 clear
    IDLE --> [*]: 长期无事件<br/>Redis 24h TTL 过期
```

## 四、扫描器工作流时序图

```mermaid
sequenceDiagram
    autonumber
    participant Sched as @Scheduled<br/>5s 触发器
    participant Scanner as PendingStateScanner<br/>pending-scanner-0
    participant Redis as Redis
    participant Repo as AlertRuleRepository
    participant Exec as AsyncAlertExecutor<br/>alert-eval-N
    participant Engine as AlertEngine
    participant Store as AlertStateStore
    participant Notifier as AlertNotifier
    participant SMTP as QQ Mail

    Sched->>Scanner: scan()
    activate Scanner
    Scanner->>Redis: SCAN alert:state:* COUNT 200
    Redis-->>Scanner: List<PendingEntry>
    loop for each entry
        Scanner->>Repo: findById(ruleId)
        alt 规则已删除
            Repo-->>Scanner: empty
            Scanner->>Store: clear(ruleId, appid) [兜底清理]
        else 规则存在
            alt 规则已禁用
                Repo-->>Scanner: rule.status=disabled
                Scanner->>Store: clear(ruleId, appid) [兜底清理]
            else 规则有效 enabled
                Repo-->>Scanner: AlertRule
                Scanner->>Exec: submitFromScanner(rule, appid, firstBreachAt, triggerCount, now)
                deactivate Scanner
            end
        end
    end

    activate Exec
    Exec->>Engine: fireFromScanner(...)
    activate Engine
    Engine->>Store: getState(ruleId, appid)
    alt state != PENDING
        Store-->>Engine: IDLE/FIRING
        Engine-->>Exec: 跳过 (状态已变)
    else state == PENDING
        Engine->>Engine: 校验 times / durationSec
        alt 未达标
            Engine-->>Exec: 跳过 (等下次)
        else 达标
            Engine->>Store: setState(FIRING, firstBreachAt, now)
            Engine->>Engine: fire(rule, synthetic, now)
            activate Notifier
            Notifier->>SMTP: send() 587 STARTTLS
            SMTP-->>Notifier: 250 OK
            Notifier-->>Engine: notifyResult
            deactivate Notifier
        end
    end
    Engine-->>Exec: 完成
    deactivate Engine
    deactivate Exec
```

## 五、典型调用时序(JVM 堆内存 > 500MB 持续 60s)

```mermaid
sequenceDiagram
    autonumber
    participant JVM as 目标 JVM
    participant Agent as AgentMetricsCollector
    participant Kafka as Kafka
    participant BAC as BatchAlertConsumer
    participant Exec as AsyncAlertExecutor
    participant Engine as AlertEngine
    participant Store as AlertStateStore
    participant Notifier as AlertNotifier
    participant SMTP as QQ Mail
    participant Scanner as PendingStateScanner<br/>5s 周期

    Note over JVM: 内存 = 520MB (超过 500MB)
    JVM->>Agent: GET :9464/metrics
    Agent->>Kafka: send MetricEvent<br/>jvm_memory_used_bytes=520MB
    Kafka->>BAC: 批量拉取 (max 500)
    BAC->>Exec: submit(MetricEvent)
    Exec->>Engine: process(MetricEvent)
    Engine->>Store: getState() → IDLE
    Engine->>Store: setState(PENDING, firstBreachAt=now)
    Note over Engine: elapsed=0, 持续时长未达

    par 后续事件持续触发
        loop 每 60s 一次,内存仍在 520MB
            Kafka->>BAC: 批量事件
            BAC->>Exec: submit
            Exec->>Engine: process
            Engine->>Store: handleBreach(PENDING, 累加 triggerCount, elapsed++)
        end
    and 扫描器兜底
        loop 每 5s 一次
            Scanner->>Store: scanPending
            Store-->>Scanner: 看到 PENDING
            Scanner->>Exec: submitFromScanner
            Exec->>Engine: fireFromScanner
            Note over Engine: 校验 elapsed < 60s,跳过
        end
    end

    Note over JVM: T+60s 仍 520MB
    Engine->>Store: getState() → PENDING
    Engine->>Store: setState(FIRING, firstBreachAt, lastFiredAt=now)
    Engine->>Notifier: notify(rule, event, "firing")
    Notifier->>SMTP: 邮件发送
    SMTP-->>Notifier: 250 OK
    Notifier-->>Engine: {"status":"ok",...}

    Note over JVM: T+120s 内存 = 480MB (恢复)
    Kafka->>BAC: jvm_memory_used_bytes=480MB
    BAC->>Exec: submit
    Exec->>Engine: process
    Engine->>Store: getState() → FIRING
    Engine->>Store: setState(RESOLVED)
    Engine->>Notifier: notify(rule, event, "resolved")
    Notifier->>SMTP: 邮件发送(恢复)
    SMTP-->>Notifier: 250 OK
    Engine->>Store: clear() [key 删除]
```

## 七、组件清单

| 文件 | 职责 | 关键方法/字段 |
|---|---|---|
| `alerter/AlertEngine` | 状态机主控 | `process(MetricEvent/LogEvent)`, `handleBreach/Recover`, `fire/resolve`, `fireFromScanner` |
| `alerter/AlertEvaluator` | 表达式求值 | `evaluate(rule, event) → BreachResult`, `simpleEvaluate`(正则), `JexlExprEvaluator`(复杂), `isLogBreached` |
| `alerter/AlertStateStore` | Redis 状态存取 | `getState/setState/clear`, `scanPending`(游标 SCAN) |
| `alerter/AlertRuleCache` | 规则内存缓存 | `rulesFor(appid)`, `@Scheduled 30s` 刷新 |
| `alerter/AlertNotifier` | SMTP 通知 | `notify(rule, event, type)`, `resolveEmail`, `sendEmail` |
| `alerter/AsyncAlertExecutor` | 异步线程池 | `submit(MetricEvent/LogEvent)`, `submitFromScanner`,虚拟线程池 8 |
| `alerter/PendingStateScanner` | **PENDING 扫描器(借鉴 HertzBeat)** | `@Scheduled 5s` 扫,过滤 PENDING,提交 fire |
| `alerter/AlertState` | 状态枚举 | `IDLE / PENDING / FIRING / RESOLVED` |
| `alerter/JexlExprEvaluator` | JEXL 引擎封装 | Apache Commons JEXL,sandbox |
| `analysis/LogAnomalyDetector` | 日志新模式检测 | `isNewPattern(appid, fingerprint)`, Redis 滑动窗口 |
| `analysis/LogAlertScheduler` | log_error_rate 定时器 | `@Scheduled 60s` 合成 MetricEvent 走 process() |

## 九、与 HertzBeat 的对应关系

```mermaid
flowchart LR
    subgraph HB[HertzBeat]
        direction TB
        HB1[PeriodicAlertRuleScheduler<br/>每规则一个调度]
        HB2[MetricsPeriodicAlertCalculator]
        HB3[AlarmCacheManager<br/>内存 Table]
        HB4[SingleAlert<br/>DB 持久化]
    end

    subgraph SW[spring-watch]
        direction TB
        SW1[PendingStateScanner<br/>5s 全局扫描]
        SW2[AlertEngine.fireFromScanner]
        SW3[AlertStateStore<br/>Redis Hash]
        SW4[AlertHistory<br/>JPA]
    end

    HB1 -->|对应| SW1
    HB2 -->|对应| SW2
    HB3 -->|对应| SW3
    HB4 -->|对应| SW4

    style SW1 fill:#dbeafe
    style SW2 fill:#dbeafe
```

**关键差异**:spring-watch 用**事件 + 扫描器双轨**驱动,HertzBeat 是**纯周期**。事件路径适合低延迟(实时反映),扫描器适合兜底(事件停流入也能告警)。两者结合比 HertzBeat 更稳。
